# 计划文档评审提示

本文档提供评审实施计划块的说明。

## 评审目的

确保每个计划块在执行前符合质量标准。评审员验证完整性、精确性和对规划原则的遵守。

## 评审检查清单

### 1. 标题完整性

- [ ] 计划有必需的标题，包含目标、架构、技术栈
- [ ] 目标是一句清晰的话
- [ ] 架构提供 2-3 句方法描述
- [ ] 技术栈列出关键技术/库

### 2. 文件结构

- [ ] 文件结构在任务前已记录
- [ ] 所有文件路径精确明确
- [ ] 新建文件、修改文件和测试文件清晰分离
- [ ] 每个文件有清晰的职责描述
- [ ] 文件扩展名与项目编程语言匹配

### 3. 任务质量

- [ ] 每个任务有清晰的文件引用（创建/修改/测试）
- [ ] 每个步骤粒度适中（每步 2-5 分钟）
- [ ] 测试先于实现（TDD 方法）
- [ ] 每个任务独立可测试

### 4. 代码完整性

- [ ] 所有代码片段完整（非占位符）
- [ ] 没有"添加验证"等模糊指令
- [ ] 代码语法正确
- [ ] 代码遵循语言约定
- [ ] 代码示例使用项目的编程语言

### 5. 命令精确性

- [ ] 所有命令精确可运行
- [ ] 预期输出已记录
- [ ] 测试命令指定确切的测试名称/路径
- [ ] 命令适合项目的语言/框架

### 6. 范围验证

- [ ] 块 ≤1000 行
- [ ] 块逻辑自洽
- [ ] 没有孤立引用其他块

## 评审输出格式

评审后，按以下格式提供输出：

```markdown
## 评审结果：[✅ 批准 / ❌ 发现问题]

### 摘要
[评审简要摘要]

### 发现的问题（如有）
1. [问题描述]
   - 位置：[具体章节/行]
   - 建议：[如何修复]

### 优点
- [做得好的地方]

### 建议
- [可选改进]
```

## 评审指导

1. **彻底但公平** - 专注于会阻碍执行的问题
2. **解释原因** - 标记问题时，解释为什么重要
3. **建设性** - 提供具体建议
4. **考虑上下文** - 计划是给不熟悉代码库的工程师看的
5. **重视精确性** - 模糊的计划导致执行失败
6. **语言意识** - 确保示例与项目编程语言匹配

## 需要注意的常见问题

### 模糊指令

❌ 错误："添加错误处理"
✅ 正确："添加 try-except 块，捕获 ValueError 并返回 ErrorResult"

### 缺少测试步骤

❌ 错误：实现步骤没有前置测试
✅ 正确：测试步骤 → 运行测试（失败）→ 实现 → 运行测试（通过）

### 不精确的路径

❌ 错误："更新配置文件"
✅ 正确："修改：`src/config/settings.py:45-52`"

### 占位符代码

❌ 错误："// 在此实现逻辑"
✅ 正确：完整、可运行的代码片段

### 错误的语言示例

❌ 错误：TypeScript 项目中使用 Python 示例
✅ 正确：TypeScript 示例与项目语言匹配

## 升级处理

如果同一块的评审循环超过 5 次迭代，升级到人工指导，提供：

1. 问题摘要
2. 已尝试的方法
3. 具体分歧领域
