## 评审目的

确保计划在各个阶段都符合质量标准，尽早发现问题，避免返工。

---

## 评审触发矩阵

| 触发点 | 评审类型 | 评审范围 | 评审重点 | 评审方式 |
|--------|---------|---------|---------|---------|
| 阶段 1 完成 | 范围评审 | 子系统识别、依赖关系 | 子系统划分、执行顺序、决策合理性 | 用户确认 |
| 阶段 2 完成 | 结构评审 | 文件结构 | 结构合理性、职责清晰、模式一致 | 用户确认 |
| 阶段 4 完成 | 任务评审 | 任务定义 | 任务完整性、粒度、TDD 流程 | 用户确认 |
| 每块完成 | 块级评审 | 当前块 | 代码完整性、测试覆盖、路径精确 | 子代理评审 |
| 子系统完成 | 子系统评审 | 整个子系统 | 整体一致性、依赖满足、共享组件 | 子代理评审 |
| 全部完成 | 最终评审 | 所有计划 | 整体质量、执行顺序、纲领完整性 | 用户确认 |
| 变更发生 | 变更评审 | 变更影响范围 | 影响分析、依赖更新、一致性 | 子代理评审 |

## 范围评审检查清单（阶段 1）

### 1. 子系统识别验证

- [ ] 子系统划分是否合理
- [ ] 是否有遗漏或冗余的子系统
- [ ] 子系统边界是否清晰
- [ ] 子系统识别标准是否满足

### 2. 依赖关系验证

- [ ] 依赖关系是否正确
- [ ] 执行顺序是否合理
- [ ] 是否有循环依赖
- [ ] 依赖分析是否完整

### 3. 决策合理性验证

- [ ] 技术选型是否合适
- [ ] 架构决策是否有依据
- [ ] 决策是否记录在纲领文件
- [ ] 决策影响范围是否明确

### 4. 纲领文件完整性

- [ ] 会话状态是否完整
- [ ] 项目概述是否清晰
- [ ] 子系统划分是否记录
- [ ] 执行顺序是否记录

---

## 结构评审检查清单（阶段 2）

### 1. 结构合理性验证

- [ ] 文件划分是否符合单一职责原则
- [ ] 文件大小是否适中（上下文友好）
- [ ] 文件命名是否清晰
- [ ] 目录结构是否合理

### 2. 职责清晰性验证

- [ ] 每个文件的职责是否明确
- [ ] 文件间是否有职责重叠
- [ ] 是否有遗漏的文件
- [ ] 测试文件是否对应源文件

### 3. 模式一致性验证

- [ ] 是否遵循现有代码库模式
- [ ] 新建/修改/测试文件是否清晰分离
- [ ] 文件扩展名是否与项目语言匹配
- [ ] 是否有共享组件需要标注

---

## 任务评审检查清单（阶段 4）

### 1. 任务完整性验证

- [ ] 任务是否覆盖所有功能点
- [ ] 任务是否有遗漏
- [ ] 任务顺序是否合理
- [ ] 任务与文件结构是否对应

### 2. 任务粒度验证

- [ ] 每个步骤是否可在 2-5 分钟内完成
- [ ] 任务是否过于复杂需要拆分
- [ ] 任务是否过于简单需要合并
- [ ] 步骤划分是否合理

### 3. TDD 流程验证

- [ ] 每个任务是否以测试开始
- [ ] 测试是否覆盖关键场景
- [ ] 实现是否是最小化
- [ ] 测试命令是否精确

### 4. 代码质量验证

- [ ] 代码片段是否完整可运行
- [ ] 命令是否精确
- [ ] 文件路径是否准确
- [ ] 代码示例是否使用项目编程语言

---

## 块级评审检查清单

### 1. 标题完整性

- [ ] 计划有必需的标题，包含目标、架构、技术栈
- [ ] 目标是一句清晰的话
- [ ] 架构提供 2-3 句方法描述
- [ ] 技术栈列出关键技术/库
- [ ] 如为子系统计划，有纲领文件路径引用

### 2. 文件结构

- [ ] 文件结构在任务前已记录
- [ ] 所有文件路径精确明确
- [ ] 新建文件、修改文件和测试文件清晰分离
- [ ] 每个文件有清晰的职责描述
- [ ] 文件扩展名与项目编程语言匹配
- [ ] 如有共享组件，已明确标注使用子系统

### 3. 任务质量

- [ ] 每个任务有清晰的文件引用（创建/修改/测试）
- [ ] 每个步骤粒度适中（每步 2-5 分钟）
- [ ] 测试先于实现（TDD 方法）
- [ ] 每个任务独立可测试

### 4. 代码完整性

- [ ] 所有代码片段完整（非占位符）
- [ ] 没有"添加验证"等模糊指令
- [ ] 代码语法正确
- [ ] 代码遵循语言约定
- [ ] 代码示例使用项目的编程语言

### 5. 命令精确性

- [ ] 所有命令精确可运行
- [ ] 预期输出已记录
- [ ] 测试命令指定确切的测试名称/路径
- [ ] 命令适合项目的语言/框架

### 6. 分块质量

- [ ] 块在 300-500 行范围内
- [ ] 块逻辑自洽
- [ ] 没有孤立引用其他块
- [ ] 块内任务有清晰的逻辑关联

## 子系统评审检查清单

### 1. 整体一致性检查

- [ ] 文件结构是否完整
- [ ] 任务是否覆盖所有功能点
- [ ] 测试是否充分
- [ ] 所有块是否通过块级评审

### 2. 依赖满足检查

- [ ] 依赖的子系统是否已完成
- [ ] 共享组件是否已定义
- [ ] 接口是否匹配
- [ ] 依赖关系是否在纲领文件中记录

### 3. 共享组件检查

- [ ] 共享组件在最早使用的子系统中定义
- [ ] 后续子系统正确引用共享组件
- [ ] 共享组件接口清晰
- [ ] 共享组件有独立测试

### 4. 纲领文件一致性

- [ ] 子系统范围与纲领描述一致
- [ ] 依赖关系正确
- [ ] 共享组件位置正确
- [ ] 执行顺序符合依赖关系

- [ ] 是否有循环依赖

### 4. 多子系统项目检查

- [ ] 所有子系统计划已完成

- [ ] 共享组件定义完整

## 变更评审检查清单

### 1. 变更完整性

- [ ] 变更已在纲领文件"变更历史"中记录
- [ ] 变更影响分析完整
- [ ] 受影响的任务已更新

### 2. 影响分析验证

- [ ] 直接影响的任务识别完整
- [ ] 间接影响的任务识别完整
- [ ] 影响的子系统识别完整

### 3. 一致性检查

- [ ] 更新后的计划与其他计划一致
- [ ] 依赖关系仍然正确
- [ ] 共享组件引用正确

### 4. 新决策记录

- [ ] 变更相关的新决策已记录
- [ ] 决策理由充分
- [ ] 影响范围明确

## 评审输出格式

评审后，按以下格式提供输出：

```markdown
## 评审结果：[✅ 批准 / ❌ 发现问题]

### 评审类型
[范围评审 / 结构评审 / 任务评审 / 块级评审 / 子系统评审 / 最终评审 / 变更评审]

### 摘要
[评审简要摘要]

### 发现的问题（如有）
1. [问题描述]
   - 位置：[具体章节/行]
   - 建议：[如何修复]

### 优点
- [做得好的地方]

### 建议
- [可选改进]
```

## 评审指导

1. **彻底但公平** - 专注于会阻碍执行的问题
2. **解释原因** - 标记问题时，解释为什么重要
3. **建设性** - 提供具体建议
4. **考虑上下文** - 计划是给不熟悉代码库的工程师看的
5. **重视精确性** - 模糊的计划导致执行失败
6. **语言意识** - 确保示例与项目编程语言匹配
7. **分块意识** - 确保每块可独立理解和执行
8. **决策一致性** - 检查是否与纲领文件中的决策记录一致
9. **尽早发现问题** - 阶段评审的目的是尽早发现问题，避免返工

## 需要注意的常见问题

### 模糊指令

❌ 错误："添加错误处理"
✅ 正确："添加 try-except 块，捕获 ValueError 并返回 ErrorResult"

### 缺少测试步骤

❌ 错误：实现步骤没有前置测试
✅ 正确：测试步骤 → 运行测试（失败）→ 实现 → 运行测试（通过）

### 不精确的路径

❌ 错误："更新配置文件"
✅ 正确："修改：`src/config/settings.py:45-52`"

### 占位符代码

❌ 错误："// 在此实现逻辑"
✅ 正确：完整、可运行的代码片段

### 错误的语言示例

❌ 错误：TypeScript 项目中使用 Python 示例
✅ 正确：TypeScript 示例与项目语言匹配

### 分块过大

❌ 错误：单个块超过 500 行
✅ 正确：将块拆分为更小的逻辑单元

### 跨块依赖未说明

❌ 错误：块中引用其他块的组件但未说明
✅ 正确：明确标注依赖关系和引用位置

### 决策未记录

❌ 错误：计划中做出技术决策但未在纲领文件中记录
✅ 正确：在纲领文件"决策记录"表格中记录决策

### 变更未追踪

❌ 错误：修改计划但未在纲领文件中记录变更
✅ 正确：在纲领文件"变更历史"表格中记录变更

### 子系统识别错误

❌ 错误：将应该拆分的功能合并为一个子系统
✅ 正确：按独立数据模型、独立接口、独立部署标准识别子系统

### 文件职责不清

❌ 错误：一个文件承担多个不相关的职责
✅ 正确：每个文件有单一明确的职责

## 升级处理

如果同一评审循环超过 5 次迭代，升级到人工指导，提供：

1. 问题摘要
2. 已尝试的方法
3. 具体分歧领域
