# 计划变更管理

处理计划编写过程中的变更请求，确保变更影响可控。

## 目录

- [变更类型](#变更类型)

## 变更类型

| 变更类型   | 说明             | 影响程度 |
| ---------- | ---------------- | -------- |
| 新增功能   | 添加新的功能需求 | 高       |
| 修改需求   | 改变现有需求     | 中-高    |
| 删除功能   | 移除已规划的功能 | 中       |
| 技术调整   | 更换技术方案     | 中-高    |
| 优先级调整 | 改变执行顺序     | 低-中    |

## 变更处理流程

### 步骤 1：变更登记

在纲领文件的"变更历史"表格中记录：

```markdown
| 变更时间   | 变更内容   | 影响范围 | 变更原因     |
| ---------- | ---------- | -------- | ------------ |
| YYYY-MM-DD | 增加缓存层 | 子系统 A | 性能优化需求 |
```

### 步骤 2：影响分析

分析变更的影响范围：

```markdown
## 变更影响分析

**变更内容**：[具体变更描述]

**直接影响的任务**：

- 任务 X：[如何影响]
- 任务 Y：[如何影响]

**间接影响的任务**：

- 任务 Z：[如何影响]

**影响的子系统**：

- 子系统 A：[如何影响]

**需要更新的文档**：

- [ ] 计划文档 A
- [ ] 纲领文件
```

### 步骤 3：更新计划

1. 更新受影响的计划文档
2. 更新纲领文件中的决策记录（如有新决策）
3. 更新子系统状态（如需要）

### 步骤 4：变更评审

对变更部分进行专项评审：

- 评审变更是否完整
- 评审影响分析是否准确
- 评审是否有遗漏

### 步骤 5：通知用户

变更完成后通知用户：

> "变更已完成。影响范围：[影响范围]。已更新文档：[文档列表]。"

## 变更决策记录

如果变更涉及新的技术决策，必须记录：

```markdown
| 决策           | 理由     | 影响范围 | 决策时间   |
| -------------- | -------- | -------- | ---------- |
| [变更相关决策] | [为什么] | [影响]   | YYYY-MM-DD |
```
