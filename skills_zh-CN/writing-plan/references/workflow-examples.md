## 工作流程示例

### 示例 1：单一子系统项目

```
用户：我需要实现使用 JWT 令牌的用户认证。这是我的规范...

代理：
1. 声明："我正在使用 writing-plan skill 来创建实施计划。"
2. 【阶段 1/7】创建纲领文件，确认单一子系统
3. 记录决策：选择 JWT 方案 | 理由：无状态、易扩展
4. 询问保存位置（默认：design/plans/）
5. 【阶段 2/7】规划文件结构
6. 【阶段 3/7】编写计划标题
7. 【阶段 4/7】引用"认证模块"模式，编写任务
8. 【阶段 5/7】分块输出，每块评审
9. 【阶段 6/7】保存计划，最终评审，请用户确认计划
10. 【阶段 7/7】询问执行意向
    - 用户选择"立即执行" → 调用 executing-plans skill
    - 用户选择"稍后执行" → 结束本技能
```

### 示例 2：多子系统项目

```
用户：我需要构建一个电商系统，包含用户管理、商品管理、订单管理。这是我的规范...

代理：
1. 声明："我正在使用 writing-plan skill 来创建实施计划。"
2. 【阶段 1/7】创建纲领文件
3. 识别 3 个子系统：用户管理、商品管理、订单管理
4. 分析依赖：订单依赖用户和商品
5. 记录决策：微服务架构 | 理由：独立部署、技术栈灵活
6. 完善纲领文件，记录执行顺序
7. 【阶段 2-5/7】处理子系统 A（用户管理）：
   - 读取纲领确认无依赖
   - 引用"CRUD 模块"模式
   - 规划文件结构
   - 编写计划
   - 块级评审 → 子系统评审
   - 更新纲领状态，记录决策
8. 【阶段 2-5/7】处理子系统 B（商品管理）：
   - 读取纲领确认无依赖
   - 规划文件结构
   - 编写计划
   - 块级评审 → 子系统评审
   - 更新纲领状态
9. 【阶段 2-5/7】处理子系统 C（订单管理）：
   - 读取纲领确认 A、B 已完成
   - 规划文件结构（引用 A、B 的共享组件）
   - 编写计划
   - 块级评审 → 子系统评审
   - 更新纲领状态
10. 【阶段 6/7】保存所有计划，最终评审，请用户确认计划
11. 【阶段 7/7】询问执行意向
    - 用户选择"立即执行" → 调用 executing-plans skill
    - 用户选择"稍后执行" → 结束本技能
```

### 示例 3：变更处理

```
用户：我需要在用户管理中增加缓存层。

代理：
1. 登记变更：增加缓存层 | 影响范围：用户管理 | 原因：性能优化
2. 影响分析：
   - 直接影响：用户查询任务
   - 间接影响：API 响应时间
3. 记录决策：使用 Redis | 理由：高性能、支持过期
4. 更新计划文档：
   - 新增缓存配置任务
   - 修改用户查询任务
5. 变更评审
6. 通知用户：变更已完成，影响用户管理模块
```

## 常见问题处理

### Q: 规格文档过大怎么办？

处理方式：

1. 在阶段 1 识别子系统边界
2. 完善纲领文件分解项目
3. 逐个子系统处理
4. 每个子系统独立进行计划编写

### Q: 子系统之间有复杂依赖怎么办？

处理方式：

1. 在纲领文件中详细记录依赖关系
2. 确定正确的执行顺序
3. 共享组件在最早使用的子系统中定义
4. 后续子系统引用已定义的共享组件

### Q: 对话中断后如何恢复？

处理方式：

1. 读取纲领文件 `design/plans/YYYY-MM-DD-<topic>-outline.md`
2. 查看"决策记录"了解已做决策
3. 声明当前阶段
4. 根据"会话状态"中的"下一步行动"继续

### Q: 用户要求修改已完成的部分？

处理方式：

1. 执行变更管理流程
2. 进行影响分析
3. 更新相关计划文档
4. 更新纲领文件记录变更和决策
5. 进行变更评审

### Q: 如何避免重复讨论已决策问题？

处理方式：

1. 每次对话开始时读取纲领文件中的"决策记录"
2. 遇到相关问题时先检查是否已有决策
3. 如有决策，引用决策记录并说明理由
4. 如需修改决策，按变更管理流程处理
