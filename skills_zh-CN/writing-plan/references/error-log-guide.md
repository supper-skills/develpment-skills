# 错误经验库维护指南

本文档说明如何维护错误经验库，错误应在技能**实际使用过程中**积累，而非预填充。

---

## 核心原则

- **动态积累**：错误经验来自真实使用场景，通过用户反馈和执行失败逐步添加
- **预防优先**：尽早读取已有错误记录，在犯错前预防，而非事后补救
- **持续更新**：每次发现新错误模式时追加记录

---

## 错误分类体系

| 分类         | 说明                         | 来源                     |
| ------------ | ---------------------------- | ------------------------ |
| **计划遗漏** | 未考虑依赖、遗漏边界情况     | writing-plan 评审发现    |
| **执行错误** | 命令错误、路径错误、逻辑错误 | executing-plans 执行失败 |
| **假设错误** | 假设了不存在的 API/文件/配置 | 用户纠正或执行失败       |
| **理解偏差** | 误解需求、误解设计文档       | 用户纠正                 |
| **规范违反** | 违反项目代码规范             | lint/test 失败           |

---

## 错误记录结构

```markdown
### YYYY-MM-DD | 分类 | 简短描述

**问题**：具体问题描述

**影响**：对项目/计划/执行的影响

**教训**：从错误中学到的具体做法

**关联**：涉及的技能和阶段（如 writing-plan > 任务定义）

**预防检查点**：未来避免类似错误需要检查的事项
```

---

## 错误经验库位置

错误经验库应放在**项目目录**下（用户的工作目录），而非技能库目录：

```
项目目录/
├── design/
│   ├── specs/          # 规格文档
│   ├── plans/          # 计划文档
│   └── context/        # 项目上下文
│       └── error-log.md  # 错误经验库（在此创建和维护）
```

---

## 在 writing-plan 中的使用

### 何时读取

在**阶段 4/7：任务定义**之前读取：

1. 检查项目是否存在 `design/context/error-log.md`
2. 如存在，读取并了解历史错误模式
3. 在任务定义时应用预防检查点

### 何时记录

发现以下情况时记录错误：

- 用户指出计划中的遗漏
- 评审发现未考虑的依赖或边界情况
- 用户纠正理解偏差

### 记录流程

1. 分析错误原因，归类到对应分类
2. 填写错误记录结构
3. 追加到错误经验库末尾
4. 更新统计表格

---

## 在 executing-plans 中的使用

### 何时读取

在**阶段一：加载并审阅计划**时读取：

1. 检查项目是否存在 `design/context/error-log.md`
2. 如存在，读取并了解历史错误模式
3. 在执行时应用预防检查点

### 何时记录

发现以下情况时记录错误：

- 执行失败（命令错误、路径错误、逻辑错误）
- 假设不存在的配置或 API
- 违反项目代码规范（lint/test 失败）

### 记录流程

1. 分析错误原因，归类到对应分类
2. 填写错误记录结构
3. 追加到错误经验库末尾
4. 更新统计表格

---

## 维护要点

1. **每次一记**：发现错误时立即记录，避免遗漏
2. **具体可操作**：教训和预防检查点要具体明确
3. **定期回顾**：在项目开始时回顾已有错误记录
4. **避免重复**：相似的错误可合并或标注关联

---

## 统计更新

每次添加新错误记录后，更新统计表格：

```markdown
| 分类     | 数量 | 最近更新时间 |
| -------- | ---- | ------------ |
| 计划遗漏 | N    | YYYY-MM-DD   |
| 执行错误 | N    | YYYY-MM-DD   |
| 假设错误 | N    | YYYY-MM-DD   |
| 理解偏差 | N    | YYYY-MM-DD   |
| 规范违反 | N    | YYYY-MM-DD   |
```
