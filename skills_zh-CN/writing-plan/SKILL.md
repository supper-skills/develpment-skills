# 编写计划技能

本技能用于创建多步骤开发任务的全面实施计划。在有规范或需求时，在直接接触任何代码之前使用。

## 触发条件

当检测到以下情况时，主动激活此技能：

- 用户提及"创建计划"、"实施计划"、"编写计划"
- 用户提及"任务分解"、"分解任务"、"拆解任务"
- 开始复杂的多步骤开发工作且有现有规范
- 用户提供需求/规范并希望在实施前进行规划
- 用户询问实现功能的详细步骤
- 如果同时触发 brainstorming，优先使用 brainstorming skill

### 被其他 skill 调用

当被 brainstorming skill 调用时：

1. 接收参数：规格文档路径
2. 跳过触发条件检查
3. 直接进入第一阶段

---

## 核心原则

### 开始声明

始终以声明开始：

> "我正在使用 writing-plan skill 来创建实施计划。"

### 目标受众假设

假设阅读此计划的工程师：

- 对我们的代码库**零了解**
- 在软件设计方面**品味存疑**
- 是**熟练开发者**但不熟悉我们的工具集或问题领域
- **不太熟悉**良好的测试设计

你的计划必须记录他们需要知道的一切。

### 核心价值观

- **DRY**（不要重复自己）- 没有冗余信息
- **YAGNI**（你不会需要它）- 只实现需要的内容
- **TDD**（测试驱动开发）- 先写测试
- **频繁提交** - 每个工作步骤后提交

## 第一阶段：上下文与范围分析

### 1.1 上下文设置

本技能应在专用工作树中运行（如可用，由 brainstorming 技能创建）。

### 1.2 范围检查

**关键**：在继续之前，评估规范是否涵盖多个独立子系统。

如果规范跨越多个独立子系统：

1. **停止并建议拆分**为单独的子项目规范
2. 每个计划应产出可运行、可测试的文件
3. 每个子系统获得自己的计划

如果未拆分，建议分解为独立计划——每个子系统一个计划。

### 1.3 询问保存位置

询问用户将计划保存到哪里：

- 默认目录：`design/plans/`
- 文件名格式：`YYYY-MM-DD-<feature-name>.md`

始终告知用户默认目录作为选项。

## 第二阶段：文件结构规划

**在定义任务前**，规划将创建或修改哪些文件及其职责。

### 2.1 设计原则

设计具有清晰边界和明确定义接口的单元：

- **单一职责**：每个文件应有单一明确职责
- **上下文友好**：能在一个上下文窗口中理解的代码
- **聚焦文件**：优先选择小而专注的文件，而非大型多功能文件
- **共置**：经常一起变更的文件应放在一起
- **按职责拆分**：按职责组织，而非按技术层

### 2.2 现有代码库模式

在现有代码库中：

- 遵循既定模式
- 不要单方面重构大文件
- 如果修改臃肿文件，**询问开发者**是否在计划中包含拆分
- 只有在获得开发者明确确认后，才将文件拆分添加到计划中

### 2.3 文件结构模板

记录计划的文件结构（根据项目语言调整文件扩展名）：

```markdown
## 文件结构

### 新建文件
- `path/to/new/file.<ext>` - 职责描述
- `path/to/another/file.<ext>` - 职责描述

### 修改文件
- `path/to/existing/file.<ext>` - 将被修改的内容
- `path/to/another/existing.<ext>` - 将被修改的内容

### 测试文件
- `tests/path/to/test_file.<ext>` - 测试职责
```

## 第四阶段：任务定义

### 4.1 任务粒度

每个步骤应对应**一个操作**（耗时 2-5 分钟）：

- "编写失败测试" - 一个步骤
- "运行测试确认失败" - 一个步骤
- "运行测试确保通过" - 一个步骤

### 4.2 任务结构模板

每个任务遵循此结构。根据项目的编程语言调整代码示例和测试命令：

```markdown
### 任务 N：[组件名称]

**文件：**
- 创建：`exact/path/to/file.<ext>`
- 修改：`exact/path/to/existing.<ext>#L123-145`
- 测试：`tests/exact/path/to/test.<ext>`

- [ ] **步骤 1：编写失败测试**

```<language>
// 示例：TypeScript
describe('functionName', () => {
  it('should return expected result for given input', () => {
    const result = functionName(input);
    expect(result).toBe(expected);
  });
});
```

- [ ] **步骤 2：运行测试验证失败**

运行：`<test-command>`（例如 `npm test`、`pytest`、`cargo test`）
预期：失败，显示适当的错误消息

- [ ] **步骤 3：编写最小实现**

```<language>
// 通过测试的最小实现
export function functionName(input: InputType): OutputType {
  return expected;
}
```

- [ ] **步骤 4：运行测试验证通过**

运行：`<test-command>`
预期：通过

### 4.3 各语言测试命令

各语言常用测试命令：

| 语言 | 测试框架 | 命令 |
|------|----------|------|
| TypeScript/JavaScript | Jest | `npm test` 或 `jest tests/path/test.ts` |
| TypeScript/JavaScript | Vitest | `vitest run tests/path/test.ts` |
| Python | pytest | `pytest tests/path/test.py::test_name -v` |
| Rust | cargo test | `cargo test test_name` |
| Go | go test | `go test ./path/to/package -run TestName` |
| Java | JUnit | `mvn test -Dtest=TestClassName` |
| C# | xUnit/NUnit | `dotnet test --filter TestName` |

### 4.4 任务要求

- **精确文件路径** - 无模糊引用
- **计划中完整代码** - 不是"添加验证"而是实际代码
- **精确命令** - 确切命令及预期输出
- **测试优先方法** - 每个任务以失败测试开始
- **语言适配** - 使用项目的编程语言编写所有示例

## 依赖管理规划

在计划中必须明确依赖管理策略，确保实施过程顺利。

### 依赖识别

**需要识别的依赖类型：**

| 依赖类型 | 说明 | 示例 |
|---------|------|------|
| 运行时依赖 | 应用运行所需的包 | `express`, `lodash`, `axios` |
| 开发依赖 | 仅开发环境需要的包 | `typescript`, `jest`, `eslint` |
| 对等依赖 | 需要使用者提供的包 | `react`, `vue` |
| 可选依赖 | 非必需但增强功能的包 | 性能分析工具、调试工具 |

### 依赖安装计划

在计划中包含以下内容：

```markdown
### 前置任务：依赖安装

**新增依赖：**
- `package-name@version` - 用途说明
- `another-package@^1.2.0` - 用途说明

**安装命令：**
- [ ] 安装运行时依赖
  运行：`pnpm add package-name@version`
  预期：依赖添加到 package.json 并安装成功

- [ ] 安装开发依赖
  运行：`pnpm add -D dev-package@version`
  预期：依赖添加到 devDependencies 并安装成功
```

### 版本管理策略

**版本锁定原则：**

- 生产环境依赖使用精确版本或锁定范围
- 开发依赖可使用较宽松的范围（`^` 或 `~`）
- 记录选择特定版本的理由

**版本兼容性检查：**

- 检查新依赖与现有依赖的兼容性
- 识别潜在的版本冲突
- 记录解决冲突的方案

## 环境配置规划

明确不同环境的配置策略，确保代码在各环境中正确运行。

### 环境类型

| 环境 | 用途 | 配置特点 |
|------|------|----------|
| 开发环境 | 本地开发调试 | 详细日志、热重载、模拟数据 |
| 测试环境 | 自动化测试 | 隔离数据、确定性配置 |
| 预发环境 | 上线前验证 | 接近生产配置、真实数据子集 |
| 生产环境 | 正式运行 | 性能优化、安全加固、精简日志 |

### 配置文件规划

**配置文件结构：**

```markdown
### 环境配置文件

**创建：**
- `.env.example` - 环境变量模板（提交到版本控制）
- `.env.development` - 开发环境配置（不提交）
- `.env.test` - 测试环境配置（可选提交）
- `.env.production` - 生产环境配置（不提交，通过 CI/CD 注入）

**配置项：**
- `DATABASE_URL` - 数据库连接地址
- `API_KEY` - 第三方服务密钥
- `LOG_LEVEL` - 日志级别
- `FEATURE_FLAGS` - 功能开关
```

### 配置任务模板

````markdown
### 任务：环境配置

**文件：**
- 创建：`.env.example`
- 创建：`src/config/index.ts`

- [ ] **步骤 1：创建环境变量模板**

```
# .env.example
DATABASE_URL=postgresql://localhost:5432/dbname
API_KEY=your-api-key-here
LOG_LEVEL=debug
```

- [ ] **步骤 2：创建配置加载模块**

```typescript
// src/config/index.ts
export const config = {
  database: {
    url: process.env.DATABASE_URL || 'postgresql://localhost:5432/dev',
  },
  api: {
    key: process.env.API_KEY,
  },
  log: {
    level: process.env.LOG_LEVEL || 'info',
  },
};
```
````

### 敏感信息处理

**安全原则：**

- 敏感信息（密钥、密码）绝不提交到版本控制
- 使用环境变量或密钥管理服务
- 在计划中明确标注哪些配置需要用户手动设置

## 回滚策略规划

为实施失败或意外情况准备回滚方案，降低风险。

### 回滚场景识别

| 场景 | 触发条件 | 回滚策略 |
|------|----------|----------|
| 依赖安装失败 | 版本冲突、网络问题 | 恢复 package.json/lock 文件 |
| 测试持续失败 | 无法修复的测试错误 | 回退到上一个通过状态 |
| 功能不兼容 | 与现有系统集成失败 | 撤销相关代码变更 |
| 性能退化 | 性能指标不达标 | 回退性能相关变更 |
| 安全问题 | 发现安全漏洞 | 立即回退并修复 |

### 回滚计划模板

在计划中包含回滚章节：

````markdown
## 回滚计划

### 检查点

| 检查点 | 完成任务 | 回滚命令 |
|--------|---------|----------|
| CP-1 | 依赖安装完成 | `git checkout package.json pnpm-lock.yaml && pnpm install` |
| CP-2 | 认证模块完成 | `git checkout src/auth/ tests/auth/` |
| CP-3 | API 集成完成 | `git checkout src/api/ tests/api/` |

### 回滚步骤

**如果实施失败需要回滚：**

1. **停止当前任务**
   - 记录失败原因
   - 保存当前进度

2. **确定回滚点**
   - 查看最近的检查点
   - 确认回滚范围

3. **执行回滚**
   ```bash
   # 回滚到指定检查点
   git reset --hard <checkpoint-commit>
   
   # 或选择性回滚文件
   git checkout <commit> -- path/to/files
   ```

4. **验证回滚**
   - 运行测试确保系统正常
   - 检查功能是否恢复

5. **记录问题**
   - 在设计文档中记录失败原因
   - 更新计划以避免重复问题
````

### Git 分支策略

**推荐分支模型：**

```
main (生产)
  └── develop (开发)
        └── feature/xxx (功能分支)
```

**回滚友好实践：**

- 每个任务完成后创建提交
- 使用描述性的提交信息
- 在关键检查点打标签
- 保持提交原子性（一个提交一个逻辑变更）

## 第五阶段：计划评审循环

### 5.1 分块边界

使用 `## Chunk N: <name>` 标题将计划分成块：

- 每个块应 ≤1000 行
- 每个块应逻辑自洽

### 5.2 评审流程

完成每个计划块后：

1. **派遣计划文档评审子代理**评审当前块
   - 提供：块内容、规范文档路径

2. **如果发现问题（❌）：**
   - 修复该代码块中的问题
   - 重新指派审核员处理该代码块
   - 重复直至获得批准（✅）

3. **如果批准（✅）：**
   - 继续处理下一部分（若为最后一部分，则移交执行）

### 5.3 评审指导

- 由制定计划的同一智能体进行修正（保留上下文）
- 若循环超过 5 次迭代，则提请人工指导
- 评审意见仅供参考——若认为反馈有误，需说明分歧理由

## 第六阶段：执行交接

### 6.1 保存计划

计划完成后：

1. 保存计划到用户指定目录（默认：`design/plans/`）
2. 使用文件名格式：`YYYY-MM-DD-<feature-name>.md`

### 6.2 交接消息

保存后，通知用户：

> "计划已完成并保存至 `design/plans/<filename>.md`。是否准备执行？"

### 6.3 执行选项

- 在当前会话中通过 executing-plans skill 执行计划
- 采用带检查点的批量执行模式以供审核

## 第七阶段：质量检查清单

### 7.1 计划完整性

- [ ] 计划标题完整，包含目标、架构、技术栈
- [ ] 文件结构在任务前已记录
- [ ] 所有文件路径精确明确
- [ ] 所有代码片段完整（非占位符）
- [ ] 所有命令精确且有预期输出

### 7.2 任务质量

- [ ] 每个任务有清晰的文件引用（创建/修改/测试）
- [ ] 每个步骤粒度适中（2-5 分钟）
- [ ] 测试先于实现
- [ ] 每个任务独立可测试

### 7.3 范围验证

- [ ] 计划覆盖单一内聚功能/子系统
- [ ] 如有多个子系统，已建议拆分
- [ ] 每个块 ≤1000 行且自洽

## 注意事项

1. **始终在开始时声明技能使用**
2. **规划期间不触碰代码** - 这是预实施阶段
3. **精确** - 精确路径、完整代码、精确命令
4. **始终测试优先** - 每个实现步骤跟随测试
5. **适当分块** - 每块 ≤1000 行以提高评审效率
6. **迭代评审** - 在进入下一块前修复问题
7. **执行前保存** - 计划必须在交接前持久化
8. **语言无关** - 所有示例应根据项目编程语言进行调整
