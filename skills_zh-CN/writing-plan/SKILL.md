# 编写计划技能

本技能用于创建多步骤开发任务的全面实施计划。在有规范或需求时，在直接接触任何代码之前使用。

## 核心约束

> ⚠️ 以下规则必须始终遵守，不可遗忘。

### 禁止行为（直到计划完成）

在计划完成前，**绝对禁止**：

- ❌ 调用任何实施技能
- ❌ 编写任何代码
- ❌ 创建或修改项目文件（计划文档除外）
- ❌ 采取任何实施行动

**无例外**：无论项目看似多简单，都必须遵循此流程。

### 纲领文件更新（强制要求，不可跳过）

> ⚠️ **强制要求**：纲领文件更新是**不可跳过的强制步骤**。忘记更新纲领文件会导致上下文丢失，必须重新询问用户已确认的信息。

- 激活技能后**必须立即创建**纲领文件
- 每次完成子系统计划后**必须更新**纲领文件
- 每轮对话开始时**必须读取**纲领文件恢复上下文
- 纲领文件是唯一的事实来源，必须保持与对话同步

**⚠️ 禁止行为**：

- ❌ 在未更新纲领文件的情况下进入下一阶段/子系统
- ❌ 跳过纲领文件直接进行对话
- ❌ 假设之前确认的信息已被记住（必须通过纲领文件确认）
- ❌ 在未更新纲领文件的情况下终止技能

### 终止条件

本技能的**唯一终止方式**是完成计划文档并通知用户准备执行。

### 阶段声明

每轮对话开始时，必须声明当前进度：

> 【计划编写 阶段 X/7】正在执行：[阶段名称]
> 当前处理：[子系统名称]（如适用）

## 目标受众假设

假设阅读此计划的工程师：

- 对我们的代码库**零了解**
- 在软件设计方面**品味存疑**
- 是**熟练开发者**但不熟悉我们的工具集或问题领域
- **不太熟悉**良好的测试设计

### 核心价值观

- **DRY**（不要重复自己）- 没有冗余信息
- **YAGNI**（你不会需要它）- 只实现需要的内容
- **TDD**（测试驱动开发）- 先写测试
- **频繁提交** - 每个工作步骤后提交

## 纲领文件机制

**目的**：解决多轮对话中的注意力丢失问题，统一管理项目信息、决策记录和进度。

> ⚠️ **强制要求**：纲领文件更新是**不可跳过的强制步骤**。每次获得用户确认或完成阶段性成果后，必须立即更新纲领文件。

**路径**：`design/plans/YYYY-MM-DD-<topic>-outline.md`

**模板**：见 [plan-template.md](references/plan-template.md)

**⚠️ 必须遵守**：

- 激活技能后 → **立即创建纲领文件**
- 每次完成子系统计划后 → **立即更新纲领文件中的子系统状态**
- 每轮对话开始时 → 读取纲领文件 → 恢复上下文（包括已做决策）→ 声明当前阶段
- 检查是否可进入下一阶段/子系统

## 阶段 1/7：上下文与范围分析

### 1.1 接收输入

从 brainstorming 技能接收：

- 规格文档路径
- 项目背景信息

### 1.2 创建纲领文件

立即创建纲领文件：`design/plans/YYYY-MM-DD-<topic>-outline.md`

### 1.3 范围检查与子系统识别

**关键**：在继续之前，评估规范是否涵盖多个独立子系统。

**子系统识别标准**：

| 特征         | 说明                     |
| ------------ | ------------------------ |
| 独立数据模型 | 子系统有独立的数据结构   |
| 独立接口     | 子系统对外提供独立 API   |
| 独立部署     | 子系统可独立部署运行     |
| 清晰边界     | 子系统间通过明确接口通信 |

**处理策略**：

- **单一子系统**：直接进入阶段 2
- **多个子系统**：进入多子系统处理流程（见 [workflow-examples.md](references/workflow-examples.md)）

### 1.4 多子系统处理流程

当识别到多个子系统时：

1. 分析子系统依赖，在纲领文件中记录
2. 完善纲领文件（子系统划分、执行顺序、共享组件）
3. 逐个子系统编写计划，每次只处理一个

### 1.5 询问保存位置

询问用户将计划保存到哪里：

- 默认目录：`design/plans/`
- 文件名格式：`YYYY-MM-DD-<feature-name>.md`

### 1.6 记录关键决策

在计划编写过程中，遇到以下情况时必须记录决策：

| 决策类型 | 示例                 |
| -------- | -------------------- |
| 技术选型 | 选择框架、库、数据库 |
| 架构决策 | 分层结构、通信方式   |
| 设计模式 | 使用哪种设计模式     |
| 实现策略 | 如何处理边界情况     |

### 1.7 范围评审

向用户展示范围分析结果并确认：

> "范围分析完成，请确认以下内容：
>
> **子系统划分**：[列出识别的子系统]
> **依赖关系**：[列出依赖关系]
> **执行顺序**：[列出执行顺序]
> **关键决策**：[列出已记录的决策]
>
> 以上内容是否正确？如有需要调整的地方，请告知。"

**完成标准**：

- 纲领文件已创建
- 子系统已识别并记录
- 保存位置已确认
- 关键决策已记录

## 阶段 2/7：文件结构规划

**在定义任务前**，规划将创建或修改哪些文件及其职责。

### 2.1 设计原则

- **单一职责**：每个文件应有单一明确职责
- **上下文友好**：能在一个上下文窗口中理解的代码
- **聚焦文件**：优先选择小而专注的文件，而非大型多功能文件
- **共置**：经常一起变更的文件应放在一起
- **按职责拆分**：按职责组织，而非按技术层

### 2.2 现有代码库模式

在现有代码库中：

- 遵循既定模式
- 不要单方面重构大文件
- 如果修改臃肿文件，**询问开发者**是否在计划中包含拆分

### 2.3 结构评审

向用户展示文件结构并确认：

> "文件结构规划完成，请确认以下内容：
>
> **新建文件**：[列出新建文件及职责]
> **修改文件**：[列出修改文件及修改内容]
> **测试文件**：[列出测试文件]
>
> 以上文件结构是否合理？如有需要调整的地方，请告知。"

## 阶段 3/7：计划文档标题

每个计划必须以此标题开头（模板见 [plan-template.md](references/plan-template.md)）：

```markdown
# [功能名称] 实施计划

> **对于代理工作者：** 必需：使用 executing-plans 来实施此计划。步骤使用复选框（`- [ ]`）语法进行跟踪。

**目标：** [一句话描述此功能构建什么]

**架构：** [2-3 句关于方法]

**技术栈：** [关键技术/库]

**所属纲领：** [纲领文件路径]（如适用）
```

## 阶段 4/7：任务定义

### 4.0 错误经验库查询

**在定义任务前，检查并读取错误经验库**：

1. 检查项目是否存在 `design/context/error-log.md`
2. 如存在，读取并了解历史错误模式
3. 根据当前任务类型，查询相关分类的错误记录
4. 将预防检查点纳入任务定义
5. **如发现新错误模式，必须追加记录到错误经验库**（见 [error-log-guide.md](references/error-log-guide.md)）

### 4.1 任务粒度

每个步骤应对应**一个操作**（耗时 2-5 分钟）。

### 4.2 任务结构模板

详见 [plan-template.md](references/plan-template.md)。

每个任务包含：

- 精确文件路径（创建/修改/测试）
- TDD 步骤：编写失败测试 → 运行测试验证失败 → 编写最小实现 → 运行测试验证通过

### 4.3 各语言测试命令

| 语言                  | 测试框架    | 命令                                      |
| --------------------- | ----------- | ----------------------------------------- |
| TypeScript/JavaScript | Jest        | `npm test` 或 `jest tests/path/test.ts`   |
| TypeScript/JavaScript | Vitest      | `vitest run tests/path/test.ts`           |
| Python                | pytest      | `pytest tests/path/test.py::test_name -v` |
| Rust                  | cargo test  | `cargo test test_name`                    |
| Go                    | go test     | `go test ./path/to/package -run TestName` |
| Java                  | JUnit       | `mvn test -Dtest=TestClassName`           |
| C#                    | xUnit/NUnit | `dotnet test --filter TestName`           |

### 4.4 任务要求

- **精确文件路径** - 无模糊引用
- **计划中完整代码** - 不是"添加验证"而是实际代码
- **精确命令** - 确切命令及预期输出
- **测试优先方法** - 每个任务以失败测试开始
- **语言适配** - 使用项目的编程语言编写所有示例

### 4.5 任务评审

向用户展示任务概览并确认：

> "任务定义完成，请确认以下内容：
>
> **任务总数**：[数量]
> **任务列表**：
>
> 1. [任务名称] - [文件]
>    ...
>
> 以上任务定义是否合理？如有需要调整的地方，请告知。"

## 阶段 5/7：分块输出与评审

### 5.1 分块策略

| 指标   | 标准       | 原因               |
| ------ | ---------- | ------------------ |
| 行数   | 300-500 行 | 适配 AI 上下文窗口 |
| 任务数 | 5-10 个    | 保持逻辑完整性     |
| 依赖   | 块内自洽   | 减少跨块引用       |

**分块边界识别**：

1. 按功能模块分块
2. 按实施阶段分块（如"基础设施"、"核心功能"、"集成测试"）
3. 按子系统分块

### 5.2 评审流程

详细评审标准见 [plan-document-reviewer-prompt.md](references/plan-document-reviewer-prompt.md)。

**评审触发矩阵**：

| 触发点      | 评审类型   | 评审方式   |
| ----------- | ---------- | ---------- |
| 阶段 1 完成 | 范围评审   | 用户确认   |
| 阶段 2 完成 | 结构评审   | 用户确认   |
| 阶段 4 完成 | 任务评审   | 用户确认   |
| 每块完成    | 块级评审   | 子代理评审 |
| 子系统完成  | 子系统评审 | 子代理评审 |
| 全部完成    | 最终评审   | 用户确认   |
| 变更发生    | 变更评审   | 子代理评审 |

### 5.3 块级评审流程

完成每个计划块后：

1. **派遣计划文档评审子代理**评审当前块
2. **如果发现问题（❌）**：修复问题，重新评审
3. **如果批准（✅）**：继续下一部分

### 5.4 多子系统场景

**⚠️ 强制要求**：每个子系统完成后，必须**立即更新纲领文件**，不可跳过此步骤。

每个子系统完成后：

1. ⚠️ **立即更新**纲领文件中的子系统状态（标记为 ✅ 已完成）
2. 确认下一个子系统的依赖已满足
3. 开始下一个子系统的计划编写

## 阶段 6/7：计划保存与确认

### 6.1 保存计划

> ⚠️ **强制要求**：保存计划时必须同步更新纲领文件。

1. 保存计划到用户指定目录（默认：`design/plans/`）
2. 使用文件名格式：`YYYY-MM-DD-<feature-name>.md`
3. ⚠️ **立即更新**纲领文件，标记当前子系统为 ✅ 已完成
4. 检查是否还有待处理的子系统或阶段

### 6.2 最终评审

执行最终评审（检查清单见 [quality-checklist.md](references/quality-checklist.md)）：

1. 整体质量检查
2. 执行顺序验证
3. 纲领完整性检查

### 6.3 用户确认计划

最终评审通过后，**必须请用户确认计划内容**：

> "计划已完成并保存。请确认以下内容：
>
> **计划文档**：`design/plans/<filename>.md`
> **关键决策**：[列出最重要的 2-3 个决策]
> **执行顺序**：[列出任务/子系统执行顺序]
>
> 请确认计划内容是否符合预期？如有需要修改的地方，请告知。"

## 阶段 7/7：执行交接

### 7.1 询问执行意向

用户确认计划后，**询问是否开始执行**：

> "计划已确认。是否现在开始执行计划？
>
> 选项：
>
> - **立即执行**：在当前会话中通过 executing-plans skill 执行计划
> - **稍后执行**：保存计划，稍后手动执行"

### 7.2 执行交接

**如果用户选择"立即执行"**：

调用 executing-plans skill，并传递以下参数：

- 计划文档路径：`design/plans/YYYY-MM-DD-<feature-name>.md`
- 纲领文件路径（如适用）：`design/plans/YYYY-MM-DD-<topic>-outline.md`
- 规格文档路径（如适用）

**调用格式**：

```
调用 executing-plans skill，并传递以下参数：
- 计划文档路径：design/plans/YYYY-MM-DD-<feature-name>.md
- 纲领文件路径：design/plans/YYYY-MM-DD-<topic>-outline.md（如适用）
- 规格文档路径：design/specs/YYYY-MM-DD-<topic>-design.md（如适用）
```

**多子系统项目**：传递所有子系统计划文档路径，按执行顺序排列。

**如果用户选择"稍后执行"**：结束本技能，告知用户后续可通过 executing-plans skill 执行计划
