# 执行计划技能

本技能用于执行书面实施计划，严格遵循文档步骤和质量检查点。技能激活后必须创建执行状态文件来跟踪进度。

## 任务状态

在执行状态文件中使用以下文本标记任务/子系统状态：

| 状态   | 说明                     |
| ------ | ------------------------ |
| 已完成 | 任务已完成，所有验证通过 |
| 进行中 | 任务正在执行中           |
| 待执行 | 任务尚未开始             |
| 阻塞   | 任务因问题被阻塞         |

---

## 阶段一：加载并审阅计划

### 1.1 前置检查

- 运行 `git branch --show-current` 确认当前分支
- 如果在 main/master 分支，**停止并询问**是否继续

### 1.2 创建执行状态文件

> ⚠️ **强制要求**：激活技能后**必须立即创建**执行状态文件

- **路径**：`design/plans/YYYY-MM-DD-<topic>-execution-state.md`
- **模板**：详见 `references/execution-state-template.md`
- 初始化任务状态表，所有任务标记为 待执行

### 1.3 批判性审阅

执行前，批判性评估计划：

**审阅检查清单**：

- [ ] 目标是否清晰且可实现？
- [ ] 所有文件路径是否精确且正确？
- [ ] 代码片段是否完整（非占位符）？
- [ ] 测试命令对项目是否准确？
- [ ] 每个步骤是否足够细粒度（2-5分钟）？
- [ ] 任务之间的依赖关系是否清晰？
- [ ] 技术栈是否适当？

### 1.4 决策点

**如果发现疑虑**：

1. 记录具体疑虑到执行状态文件
2. 在继续之前与用户讨论
3. 等待计划修订或澄清

**如果无疑虑**：

1. 从计划任务创建任务列表
2. 声明进入阶段二

---

## 阶段二：执行任务

### 2.1 任务执行模式

> ⚠️ **强制要求**：每完成一个任务后，必须**立即更新执行状态文件**，不可跳过此步骤

对于计划中的每个任务：

1. 在任务列表中将任务标记为 进行中
2. 完全按照书面执行每个步骤
3. 运行指定的验证
4. 在任务列表中将任务标记为 已完成
5. 【可选】调用 code-commentator 技能添加注释
6. ⚠️ 立即更新执行状态文件
7. 如果计划指定，则提交

### 2.2 步骤执行规则

- **修改前先读取** - 在进行更改之前始终读取现有文件
- **仅使用完整代码** - 使用计划中的确切代码
- **精确运行命令** - 按计划中指定的方式执行命令
- **验证输出** - 检查实际输出是否与预期输出匹配

### 2.3 验证要求

每个任务在进入下一个之前必须通过其验证：

- 测试必须通过（不仅仅是运行）
- 构建必须成功
- Lint 必须通过
- 任何指定的检查必须通过

### 2.4 错误处理

**错误发生时**：

1. 检查项目是否存在 `design/context/error-log.md`，如不存在则创建
2. 根据错误类型分类（执行错误/假设错误/规范违反）
3. 参考 `references/error-log-guide.md` 记录错误教训

### 2.5 进度持久化与恢复

**恢复前验证检查**：

| 验证项         | 命令/方法        | 目的           |
| -------------- | ---------------- | -------------- |
| Git 状态       | `git status`     | 确认工作区状态 |
| 已完成任务代码 | 检查文件是否存在 | 确认代码已写入 |
| 已完成任务测试 | 运行相关测试命令 | 确认功能正常   |

**检测到中断后，向用户报告恢复选项**：

```
检测到计划执行中断：
- 计划：[计划路径]
- 状态文件：[状态文件路径]
- 已完成：[已完成数量]（共 [总数] 个任务）
- 中断位置：[中断任务名称]
- 代码验证：[验证结果]

请选择恢复方式：
a) 继续执行 - 从中断位置继续执行后续任务
b) 重新执行 - 重新执行中断的任务
c) 从头执行 - 清除进度，从头开始执行所有任务"
```

详细恢复流程见 `references/workflow-examples.md`。

---

## 阶段三：完成与报告

### 3.1 任务完成

当所有任务完成时：

1. 运行最终验证套件
2. 更新执行状态文件标记完成
3. 总结已实施的内容
4. 列出与计划的任何偏差（如有）

### 3.2 完成报告

使用 `references/completion-report-template.md` 中的模板生成报告。

## 何时停止并寻求帮助

**在以下情况下立即停止执行**：

| 情况             | 行动                   |
| ---------------- | ---------------------- |
| 缺少依赖项       | 请求用户安装或提供     |
| 测试反复失败     | 调试并在卡住时寻求指导 |
| 指令不明确       | 请求澄清，不要猜测     |
| 计划存在关键缺陷 | 返回阶段一进行计划修订 |
| 出现意外错误     | 记录错误并寻求帮助     |

**不要强行突破阻碍** - 停止并询问

### 停止协议

1. 清楚地记录阻碍到执行状态文件
2. 解释您尝试了什么
3. 提出具体问题
4. 更新执行状态文件中的"遇到的问题"
5. **分析错误原因并追加记录到错误经验库**：
   - 检查项目是否存在 `design/context/error-log.md`，如不存在则创建
   - 根据错误类型分类
   - 参考 `references/error-log-guide.md` 进行记录
6. 在继续之前等待指导

## 何时重新审视阶段一

**在以下情况下返回计划审阅**：

- 用户根据反馈更新了计划
- 基本方法需要重新思考
- 多个任务被同一问题阻碍
- 尽管遵循计划，验证仍持续失败

---

## 多子系统处理流程

当计划包含多个子系统时，按依赖顺序逐个执行。

**流程**：

1. 读取纲领文件，识别子系统数量
2. 验证依赖关系，按执行顺序排列
3. 为整个项目和每个子系统创建状态文件
4. 按依赖顺序执行每个子系统（阶段一 → 阶段二 → 阶段三）
5. 共享组件在第一个使用它的子系统中实现

详细示例见 `references/workflow-examples.md` 中的"示例四"。

## 核心约束（必须始终遵守）

> ⚠️ 以下规则是**不可遗忘的强制要求**。

### 禁止行为

- ❌ 跳过批判性审阅（阶段一）
- ❌ 跳过验证步骤
- ❌ 在主分支上操作（除非获得明确批准）
- ❌ 猜测或假设

### 执行状态文件更新

**⚠️ 强制要求，不可跳过**：

1. 激活技能后**必须立即创建**执行状态文件
2. 每完成一个任务后**必须立即更新**执行状态文件
3. 每轮对话开始时**必须读取**执行状态文件恢复上下文

**禁止行为**：

- ❌ 在未更新执行状态文件的情况下继续执行任务
- ❌ 假设之前确认的信息已被记住（必须通过执行状态文件确认）

### 阶段声明

每轮对话开始时，必须声明当前进度：

```
【执行阶段 X/3】正在执行：[阶段名称]
当前处理：[子系统名称]（如适用）
```

## 质量检查清单

### 执行前

- [ ] 计划文件已定位且可读
- [ ] 批判性审阅已完成
- [ ] 疑虑已解决（如有）
- [ ] 已从计划创建待办列表
- [ ] 执行状态文件已创建
- [ ] 不在主分支上（或已获得明确批准）

### 执行中

- [ ] 每个步骤完全按书面执行
- [ ] 验证已运行并通过
- [ ] 任务列表实时更新
- [ ] **执行状态文件实时更新**（强制）
- [ ] 按指定进行提交

### 执行后

- [ ] 所有任务标记为 已完成
- [ ] 执行状态文件已更新为完成状态
- [ ] 最终验证通过
- [ ] 完成报告已生成
- [ ] 无未报告的偏差

## 被其他 Skill 调用

### 从 writing-plan 接收参数

当被 writing-plan 技能调用时：

1. **接收参数**：
   - `plan_path`：计划文档路径（必需）
   - `subsystem_paths`：子系统计划路径列表（多子系统时必需）
   - `outline_path`：纲领文件路径（如适用）
   - `spec_path`：规格文档路径（如适用）

2. **跳过触发条件检查**，直接创建执行状态文件并进入阶段一
