# 执行计划技能

本技能用于执行书面实施计划，严格遵循文档步骤和质量检查点。

## 参考文档索引

| 文档 | 用途 | 读取时机 |
|------|------|---------|
| `references/execution-state-template.md` | 执行状态文件完整模板 | 创建状态文件时参考 |
| `references/workflow-examples.md` | 详细工作流程示例 | 需要参考具体执行示例时 |
| `references/completion-report-template.md` | 完成报告模板 | 阶段三生成报告时参考 |
| `references/error-log-guide.md` | 错误经验库维护指南 | 读取/更新错误经验库时参考 |

---

## 触发条件

在以下情况下激活此技能：

- 用户提及"执行计划"、"实施计划"、"运行计划"
- 用户提供准备实施的计划文档
- 用户要求基于现有计划开始实施
- 用户在计划创建后说"开始吧"
- 被 writing-plan 技能调用（计划已完成并获用户批准）

---

## 被其他 skill 调用

当被 writing-plan 技能调用时：

1. **接收参数**：
   - 计划文档路径（必需）
   - 子系统计划路径列表（多子系统时必需）：按执行顺序排列
   - 纲领文件路径、规格文档路径（如适用）
2. 跳过触发条件检查
3. 创建执行状态文件
4. 进入阶段一

---

## 核心约束

> ⚠️ 以下规则必须始终遵守，不可遗忘。

### 禁止行为

- ❌ 跳过批判性审阅
- ❌ 跳过验证步骤
- ❌ 在主分支上操作（除非获得明确批准）
- ❌ 猜测或假设

### 阶段声明

每轮对话开始时，必须声明当前进度：

> 【执行阶段 X/3】正在执行：[阶段名称]
> 当前处理：[子系统名称]（如适用）

## 执行状态文件机制

**目的**：解决多轮对话中的注意力丢失问题。

**操作**：技能激活后，立即创建执行状态文件：

- **路径**：`design/plans/YYYY-MM-DD-<topic>-execution-state.md`
- **作用**：记录会话状态、任务进度、遇到的问题、上下文记录

**模板**：详见 `references/execution-state-template.md`

---

## 多子系统处理流程

> 当计划包含多个子系统时，按依赖顺序逐个执行。详见 `references/workflow-examples.md` 中的"示例四"。

### 0.1 识别子系统数量

- 读取纲领文件，检查子系统列表
- **单一子系统**：跳过本章，直接进入阶段一
- **多子系统**：按本章流程执行

### 0.2 验证依赖关系

- 读取纲领文件中的"子系统依赖分析"
- 按执行顺序排列子系统（不修改纲领文件）

### 0.3 创建执行状态文件

为整个项目和每个子系统创建状态文件。模板见 `references/execution-state-template.md`。

### 0.4 按依赖顺序执行子系统

```
对于每个子系统（按执行顺序）：
1. 检查依赖是否满足，读取依赖子系统的状态文件
2. 为该子系统创建执行状态文件
3. 进入阶段一（加载并审阅计划）
4. 进入阶段二（执行任务）
5. 标记子系统完成，更新顶层状态文件
6. 继续下一个子系统
```

### 0.5 共享组件处理

- 读取纲领文件中的"共享组件"信息
- 在第一个使用该组件的子系统中实现
- 后续子系统直接引用已实现的共享组件

### 0.6 多子系统恢复执行

1. 读取顶层执行状态文件
2. 识别中断的子系统（状态为"🔄 进行中"或"⏳ 待执行"）
3. 检查每个已完成子系统的状态文件
4. 验证代码和测试是否正常
5. 验证依赖关系后继续执行

## 阶段一：加载并审阅计划

### 1.1 前置检查

- 运行 `git branch --show-current` 确认当前分支
- 如果在 main/master 分支：停止并询问是否继续

### 1.2 接收参数与定位文件

- 使用提供的计划文档路径或询问用户
- 常见位置：`design/plans/`
- 读取纲领文件获取全局约束和共享组件信息

### 1.3 创建执行状态文件

创建 `design/plans/YYYY-MM-DD-<topic>-execution-state.md`，填写元信息和初始状态。

### 1.4 批判性审阅

执行前，批判性评估计划：

**审阅检查清单：**

- [ ] 目标是否清晰且可实现？
- [ ] 所有文件路径是否精确且正确？
- [ ] 代码片段是否完整（非占位符）？
- [ ] 测试命令对项目是否准确？
- [ ] 每个步骤是否足够细粒度（2-5分钟）？
- [ ] 任务之间的依赖关系是否清晰？
- [ ] 技术栈是否适当？

### 1.5 决策点

**如果发现疑虑：**

1. 记录具体疑虑到执行状态文件
2. 在继续之前与人类伙伴讨论
3. 等待计划修订或澄清

**如果无疑虑：**

1. 从计划任务创建任务列表
2. 继续执行

---

## 阶段二：执行任务

### 2.1 任务执行模式

对于计划中的每个任务：

```
1. 在任务列表中将任务标记为进行中
2. 完全按照书面执行每个步骤
3. 运行指定的验证
4. 在任务列表中将任务标记为已完成
5. 【可选】评估是否为代码添加注释（见 2.1.1）
6. 更新执行状态文件
7. 如果计划指定，则提交（频繁提交）
```

**错误发生时**：
- 检查项目是否存在 `design/context/error-log.md`，如不存在则创建
- 根据错误类型分类（执行错误/假设错误/规范违反）
- 分析错误原因并追加记录教训到错误经验库
- 参考 [error-log-guide.md](references/error-log-guide.md) 进行记录

### 2.1.1 注释补充（可选）

**前提条件**：项目中存在 `code-commentator` 技能。

**评估标准**：

| 情况 | 建议 |
|------|------|
| 复杂业务逻辑、关键算法、公共 API、数据模型 | ✅ 添加注释 |
| 简单工具函数、样板代码/配置 | ❌ 跳过 |

**调用方式**：

```
调用 code-commentator 技能，并传递以下参数：
- 文件列表：[本次任务创建/修改的代码文件]
- 范围：仅添加注释，不修改代码
```

### 2.2 步骤执行规则

- **修改前先读取** - 在进行更改之前始终读取现有文件
- **仅使用完整代码** - 使用计划中的确切代码
- **精确运行命令** - 按计划中指定的方式执行命令
- **验证输出** - 检查实际输出是否与预期输出匹配
- **更新状态** - 每完成一步，立即更新执行状态文件

### 2.3 验证要求

每个任务在进入下一个之前必须通过其验证：

- 测试必须通过（不仅仅是运行）
- 构建必须成功
- Lint 必须通过
- 任何指定的检查必须通过

### 2.4 进度持久化与恢复

#### 进度追踪机制

执行状态文件中的任务进度：

```markdown
| 任务 | 状态 | 完成时间 | 备注 |
|------|------|---------|------|
| 任务 1: [名称] | ✅ 完成 | HH:mm | - |
| 任务 2: [名称] | 🔄 进行中 | - | 步骤 2/4 |
| 任务 3: [名称] | ⏳ 待执行 | - | - |
```

#### 恢复执行流程

**当会话中断后恢复执行时**：

1. 读取执行状态文件，恢复上下文
2. 读取计划文档，扫描任务标记状态
3. 识别中断位置
4. 执行代码验证：
   - 检查 Git 状态：确认工作区状态
   - 检查已完成任务的代码是否存在
   - 运行相关测试验证已完成任务正常

5. 向用户报告恢复选项：

   > "检测到计划执行中断：
   > - 计划：[计划路径]
   > - 状态文件：[状态文件路径]
   > - 已完成：[已完成数量]（共 [总数] 个任务）
   > - 中断位置：[中断任务名称]
   > - 代码验证：[验证结果]
   >
   > 请选择恢复方式：
   > a) 继续执行 - 从中断位置继续执行后续任务
   > b) 重新执行 - 重新执行中断的任务
   > c) 从头执行 - 清除进度，从头开始执行所有任务"

6. 根据用户选择执行相应操作

#### 恢复前验证检查

| 验证项 | 命令/方法 | 目的 |
|--------|----------|------|
| Git 状态 | `git status` | 确认工作区状态 |
| 已完成任务代码 | 检查文件是否存在 | 确认代码已写入 |
| 已完成任务测试 | 运行相关测试命令 | 确认功能正常 |
| 分支状态 | `git branch` | 确认在正确分支 |
| 执行状态文件 | 读取状态文件 | 确认上下文正确 |

## 阶段三：完成与报告

### 3.1 所有任务完成

当所有任务完成时：

1. 运行最终验证套件
2. 更新执行状态文件标记完成
3. 总结已实施的内容
4. 列出与计划的任何偏差（如有）
5. 报告完成状态

### 3.2 完成报告

报告模板见 `references/completion-report-template.md`。

## 何时停止并寻求帮助

**在以下情况下立即停止执行**：

| 情况 | 行动 |
|------|------|
| 缺少依赖项 | 请求用户安装或提供 |
| 测试反复失败 | 调试并在卡住时寻求指导 |
| 指令不明确 | 请求澄清，不要猜测 |
| 计划存在关键缺陷 | 返回阶段一进行计划修订 |
| 出现意外错误 | 记录错误并寻求帮助 |
| 检测到范围蔓延 | 在扩展前与用户讨论 |

### 停止协议

1. 清楚地记录阻碍到执行状态文件
2. 解释您尝试了什么
3. 提出具体问题
4. 更新执行状态文件中的"遇到的问题"
5. **分析错误原因并追加记录到错误经验库**：
   - 检查项目是否存在 `design/context/error-log.md`，如不存在则创建
   - 根据错误类型分类（执行错误/假设错误/规范违反）
   - 记录错误教训和预防检查点
   - 参考 [error-log-guide.md](references/error-log-guide.md) 进行记录
6. 在继续之前等待指导

## 何时重新审视阶段一

**在以下情况下返回计划审阅**：

- 人类伙伴根据您的反馈更新了计划
- 基本方法需要重新思考
- 多个任务被同一问题阻碍
- 尽管遵循计划，验证仍持续失败

**不要强行突破阻碍** - 停止并询问

## 质量检查清单

### 执行前

- [ ] 计划文件已定位且可读
- [ ] 批判性审阅已完成
- [ ] 疑虑已解决（如有）
- [ ] 已从计划创建待办列表
- [ ] 执行状态文件已创建
- [ ] 不在主分支上（或已获得明确批准）

### 执行中

- [ ] 每个步骤完全按书面执行
- [ ] 验证已运行并通过
- [ ] 任务列表实时更新
- [ ] 执行状态文件实时更新
- [ ] 按指定进行提交

### 执行后

- [ ] 所有任务标记为完成
- [ ] 执行状态文件已更新为完成状态
- [ ] 最终验证通过
- [ ] 完成报告已生成
- [ ] 无未报告的偏差

## 注意事项

1. **始终在开始时声明技能使用**
2. **阶段声明是强制性的** - 每轮对话开始时声明当前阶段
3. **批判性审阅是强制性的** - 永远不要跳过阶段一
4. **状态文件必须维护** - 实时更新执行状态文件
5. **遇到阻碍即停止** - 询问比猜测更好
6. **精确遵循** - 不要即兴发挥计划步骤
7. **报告偏差** - 如果必须偏离，记录原因到执行状态文件
8. **频繁提交** - 如果指定，在每个工作步骤后提交
9. **未经同意不在主分支操作** - 需要明确批准
