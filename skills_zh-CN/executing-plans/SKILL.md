# 执行计划技能

本技能用于执行书面实施计划，严格遵循文档步骤和质量检查点。

## 触发条件

在以下情况下激活此技能：

- 用户提及"执行计划"、"实施计划"、"运行计划"
- 用户提供准备实施的计划文档
- 用户要求基于现有计划开始实施
- 用户在计划创建后说"开始吧"
- 被 writing-plan 技能调用（计划已完成并获用户批准）

## 被其他 skill 调用

当被 writing-plan 技能调用时：

1. 接收参数：规格文档路径
2. 跳过触发条件检查
3. 直接进入第一阶段

---

## 核心原则

### 开始声明

始终以以下声明开始：

> "我正在使用 executing-plans skill 来实施此计划。"

### 核心价值观

- **先批判性审阅** - 永远不要盲目执行；评估计划
- **严格遵循** - 完全按照书面步骤执行
- **不跳过验证** - 运行所有指定的验证
- **遇到阻碍即停止** - 寻求帮助而非猜测
- **未经同意不在主分支操作** - 未经用户明确同意，切勿在主分支上开始实施

## 阶段一：加载并审阅计划

### 1.0 前置检查

- 运行 `git branch --show-current` 确认当前分支
- 如果在 main/master 分支：
  - 停止并询问用户是否继续
  - 建议创建功能分支

### 1.1 定位计划文件

- 如果未提供，询问用户计划文件位置
- 常见位置：`design/plans/`
- 文件格式：通常为 `.md` 文件

### 1.2 批判性审阅

执行前，批判性评估计划：

**审阅检查清单：**

- [ ] 目标是否清晰且可实现？
- [ ] 所有文件路径是否精确且正确？
- [ ] 代码片段是否完整（非占位符）？
- [ ] 测试命令对项目是否准确？
- [ ] 每个步骤是否足够细粒度（2-5分钟）？
- [ ] 任务之间的依赖关系是否清晰？
- [ ] 技术栈是否适当？

### 1.3 决策点

**如果发现疑虑：**

1. 记录具体疑虑
2. 在继续之前与人类伙伴讨论
3. 等待计划修订或澄清

**如果无疑虑：**

1. 从计划任务创建待办事项列表
2. 继续执行

## 阶段二：执行任务

### 2.1 任务执行模式

对于计划中的每个任务：

```
1. 在待办列表中将任务标记为进行中
2. 完全按照书面执行每个步骤
3. 运行指定的验证
4. 将任务标记为已完成
5. 如果计划指定，则提交（频繁提交）
```

### 2.2 步骤执行规则

- **修改前先读取** - 在进行更改之前始终读取现有文件
- **仅使用完整代码** - 使用计划中的确切代码，而非近似值
- **精确运行命令** - 按计划中指定的方式执行命令
- **验证输出** - 检查实际输出是否与预期输出匹配

### 2.3 验证要求

每个任务在进入下一个之前必须通过其验证：

- 测试必须通过（不仅仅是运行）
- 构建必须成功
- Lint 必须通过
- 任何指定的检查必须通过

### 2.4 进度持久化与恢复

为确保执行过程可中断、可恢复，利用计划文档中的任务标记追踪进度。

#### 进度追踪机制

**计划文档中的任务标记**：

```markdown
- [ ] **步骤 1：编写失败测试**    ← 未完成
- [x] **步骤 2：运行测试验证失败** ← 已完成
- [x] **步骤 3：编写最小实现**    ← 已完成
- [ ] **步骤 4：运行测试验证通过** ← 未完成（中断点）
```

**中断位置识别**：
- 扫描计划文档中的任务复选框
- 第一个 `- [ ]` 标记即为中断位置
- 已完成的任务标记为 `- [x]`

#### 恢复执行流程

**当会话中断后恢复执行时**：

```
1. 读取计划文档，扫描任务标记状态
2. 识别中断位置（第一个未完成的任务）
3. 执行代码验证：
   - 检查 Git 状态：确认工作区是否干净
   - 检查已完成任务的代码是否存在
   - 运行相关测试验证已完成任务正常

4. 向用户报告恢复选项：

   > "检测到计划执行中断：
   > - 计划：design/plans/2024-01-15-auth.md
   > - 已完成：任务 1-3（共 8 个任务）
   > - 中断位置：任务 4 - 运行测试验证通过
   > - 代码验证：✅ 已完成任务的代码存在且测试通过
   > 
   > 请选择恢复方式：
   > a) 继续执行 - 从中断位置继续执行后续任务
   > b) 重新执行 - 重新执行中断的任务（当前任务 4）
   > c) 从头执行 - 清除进度，从头开始执行所有任务"

5. 根据用户选择执行相应操作
```

#### 恢复验证检查

**恢复前必须验证**：

| 验证项 | 命令/方法 | 目的 |
|--------|----------|------|
| Git 状态 | `git status` | 确认工作区状态 |
| 已完成任务代码 | 检查文件是否存在 | 确认代码已写入 |
| 已完成任务测试 | 运行相关测试命令 | 确认功能正常 |
| 分支状态 | `git branch` | 确认在正确分支 |

**验证结果处理**：

- **验证通过** → 提供继续/重新执行选项
- **验证失败** → 建议从失败的任务重新执行
- **代码缺失** → 建议从头执行或检查 Git 历史

#### 用户选择处理

**选择 a) 继续执行**：
- 保持已完成任务的状态
- 从中断位置继续执行后续任务

**选择 b) 重新执行中断任务**：
- 不修改已完成任务的代码
- 重新执行当前中断的任务

**选择 c) 从头执行**：
- 将计划文档中所有 `- [x]` 重置为 `- [ ]`
- 从第一个任务开始重新执行

## 阶段三：完成与报告

### 3.1 所有任务完成

当所有任务完成时：

1. 运行最终验证套件
2. 总结已实施的内容
3. 列出与计划的任何偏差（如有）
4. 报告完成状态

### 3.2 完成报告模板

```markdown
## 实施完成

**计划**：[计划文件路径]
**状态**：✅ 所有任务已完成

### 摘要
- [已构建内容的简要描述]

### 变更文件
- 新建：[新文件列表]
- 修改：[修改文件列表]

### 验证结果
- 测试：✅ 通过
- 构建：✅ 成功
- Lint：✅ 干净

### 备注
- [任何观察或建议]
```

## 何时停止并寻求帮助

**在以下情况下立即停止执行：**

| 情况 | 行动 |
|------|------|
| 缺少依赖项 | 请求用户安装或提供 |
| 测试反复失败 | 调试并在卡住时寻求指导 |
| 指令不明确 | 请求澄清，不要猜测 |
| 计划存在关键缺陷 | 返回阶段一进行计划修订 |
| 出现意外错误 | 记录错误并寻求帮助 |
| 检测到范围蔓延 | 在扩展前与用户讨论 |

### 停止协议

1. 清楚地记录阻碍
2. 解释您尝试了什么
3. 提出具体问题
4. 在继续之前等待指导

---

## 何时重新审视阶段一

**在以下情况下返回计划审阅：**

- 人类伙伴根据您的反馈更新了计划
- 基本方法需要重新思考
- 多个任务被同一问题阻碍
- 尽管遵循计划，验证仍持续失败

**不要强行突破阻碍** - 停止并询问

---

## 质量检查清单

### 执行前

- [ ] 计划文件已定位且可读
- [ ] 批判性审阅已完成
- [ ] 疑虑已解决（如有）
- [ ] 已从计划创建待办列表
- [ ] 不在主分支上（或已获得明确批准）

### 执行中

- [ ] 每个步骤完全按书面执行
- [ ] 验证已运行并通过
- [ ] 待办事项实时更新
- [ ] 按指定进行提交

### 执行后

- [ ] 所有任务标记为完成
- [ ] 最终验证通过
- [ ] 完成报告已生成
- [ ] 无未报告的偏差

## 工作流程示例

### 示例一：标准执行

```
用户：执行 design/plans/2024-01-15-auth.md 中的计划

代理：
1. 声明："我正在使用 executing-plans 技能来实施此计划。"
2. 告知子代理偏好
3. 从 design/plans/2024-01-15-auth.md 加载计划
4. 批判性审阅计划 - 未发现疑虑
5. 创建包含 8 个任务的待办列表
6. 执行每个任务：
   - 任务 1：编写测试 → 运行 → 失败 ✓
   - 任务 1：编写代码 → 运行 → 通过 ✓
   - 任务 2：编写测试 → 运行 → 失败 ✓
   - ...（继续）
7. 所有任务完成
8. 运行最终验证
9. 报告完成并附带摘要
```

### 示例二：计划疑虑

```
用户：执行 plans/feature-x.md 中的计划

代理：
1. 声明："我正在使用 executing-plans 技能来实施此计划。"
2. 加载计划
3. 审阅并发现疑虑：
   - 文件路径引用不存在的目录
   - 测试命令使用错误的框架
4. 停止并询问用户：
   "我在计划中发现问题：
    - 路径 'src/lib/' 不存在。应该是 'src/utils/' 吗？
    - 计划使用 'jest' 但项目使用 'vitest'
    我应该继续修正还是等待计划修订？"
5. 等待用户指导
```

### 示例三：执行中被阻碍

```
用户：继续执行计划

代理：
1. 执行任务 3
2. 测试意外失败
3. 调试发现缺少依赖
4. 停止并报告：
   "在任务 3 被阻碍：测试需要 'lodash' 但未安装。
    我应该：
    a) 将 lodash 添加到依赖项
    b) 等待您安装
    c) 修改方法"
5. 等待指导
```

## 注意事项

1. **始终在开始时声明技能使用**
2. **批判性审阅是强制性的** - 永远不要跳过阶段一
3. **遇到阻碍即停止** - 询问比猜测更好
4. **精确遵循** - 不要即兴发挥计划步骤
5. **报告偏差** - 如果必须偏离，记录原因
6. **频繁提交** - 如果指定，在每个工作步骤后提交
7. **未经同意不在主分支操作** - 需要明确批准
