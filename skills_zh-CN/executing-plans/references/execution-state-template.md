# 执行状态文件模板

本文档提供执行状态文件的模板结构，用于解决多轮对话中的注意力丢失问题。

---

## 状态文件路径

### 单一子系统

```
design/plans/YYYY-MM-DD-<topic>-execution-state.md
```

### 多子系统项目

```
顶层状态文件：design/plans/YYYY-MM-DD-<topic>-execution-state.md
│
├── 子系统 A 状态：design/plans/YYYY-MM-DD-<topic>-subsystem-a-execution.md
├── 子系统 B 状态：design/plans/YYYY-MM-DD-<topic>-subsystem-b-execution.md
└── 子系统 C 状态：design/plans/YYYY-MM-DD-<topic>-subsystem-c-execution.md
```

---

## 完整模板

### 单一子系统模板

```markdown

- **计划文档**: [计划文档路径]
- **纲领文件**: [纲领文件路径]（如适用）
- **规格文档**: [规格文档路径]（如适用）
- **开始时间**: YYYY-MM-DD HH:mm
- **最后更新**: YYYY-MM-DD HH:mm

## 执行阶段

阶段 X/3: [阶段名称]

## 当前进度

- **当前任务**: [任务编号] - [任务名称]
- **当前步骤**: [步骤编号]
- **下一步行动**: [具体行动]

| 任务           | 状态   | 完成时间 | 备注     |

| ---------- | ---------- | -------- |
```

## 字段说明

### 元信息

| 字段       | 说明                           | 必填           |
| ---------- | ------------------------------ | -------------- |
| 计划文档   | 指向正在执行的计划文档路径     | 是             |
| 纲领文件   | 指向关联的纲领文件路径（只读） | 多子系统时必填 |
| 规格文档   | 指向原始规格文档路径           | 否             |
| 子系统数量 | 项目包含的子系统数量           | 多子系统时必填 |
| 开始时间   | 执行开始的时间戳               | 是             |
| 最后更新   | 最后一次更新状态文件的时间     | 自动           |

### 执行阶段

| 阶段     | 说明           |
| -------- | -------------- |
| 阶段 1/3 | 加载并审阅计划 |
| 阶段 2/3 | 执行任务       |
| 阶段 3/3 | 完成与报告     |

### 任务状态

| 状态   | 说明                     |
| ------ | ------------------------ |
| 已完成 | 任务已完成，所有验证通过 |
| 进行中 | 任务正在执行中           |
| 待执行 | 任务尚未开始             |
| 阻塞   | 任务因问题被阻塞         |

### 子系统状态

| 状态   | 说明                       |
| ------ | -------------------------- |
| 已完成 | 子系统所有任务已完成       |
| 进行中 | 子系统正在执行中           |
| 待执行 | 子系统尚未开始（等待依赖） |
| 阻塞   | 子系统因问题被阻塞         |

## 使用说明

### 单一子系统

#### 技能激活时

1. 创建执行状态文件
2. 填写元信息（计划文档、开始时间等）
3. 设置执行阶段为"阶段 1/3: 加载并审阅计划"
4. 初始化任务进度表（所有任务状态为"待执行"）

#### 每轮对话开始时

1. 读取执行状态文件
2. 根据状态文件恢复上下文
3. 声明当前阶段

#### 每完成一个任务后

1. 更新任务进度表（状态改为"已完成"）
2. 记录完成时间
3. 更新当前任务为下一个任务
4. 更新顶层状态文件（如有）

### 多子系统

#### 技能激活时

1. 读取纲领文件，识别子系统
2. 创建顶层执行状态文件
3. 为每个子系统创建独立的执行状态文件
4. 填写元信息，设置所有子系统状态为"待执行"

#### 子系统执行流程

1. 检查依赖是否满足
2. 更新子系统状态为"进行中"
3. 更新顶层状态文件的当前子系统
4. 按阶段一 → 阶段二 → 阶段三执行该子系统
5. 子系统完成后，更新子系统状态为"已完成"
6. 更新顶层状态文件的子系统进度
7. 继续下一个子系统

#### 恢复执行时

1. 读取顶层执行状态文件
2. 识别中断的子系统
3. 检查已完成子系统的状态文件
4. 验证依赖关系
5. 从中断位置继续执行

## 与其他技能的状态文件对比

| 技能            | 状态文件     | 路径格式                                             | 用途             |
| --------------- | ------------ | ---------------------------------------------------- | ---------------- |
| brainstorming   | 状态文件     | `design/specs/YYYY-MM-DD-<topic>-state.md`           | 头脑风暴阶段跟踪 |
| writing-plan    | 纲领文件     | `design/plans/YYYY-MM-DD-<topic>-outline.md`         | 计划编写阶段跟踪 |
| executing-plans | 执行状态文件 | `design/plans/YYYY-MM-DD-<topic>-execution-state.md` | 执行阶段跟踪     |

---

## 注意事项

1. **不修改纲领文件** - executing-plans 只读取 writing-plan 的纲领文件，不做任何修改
2. **状态文件独立存储** - 所有执行状态信息存储在 executing-plans 自己的状态文件中
3. **子系统和顶层分离** - 多子系统项目使用顶层状态文件 + 子系统独立状态文件
4. 执行状态文件应与计划文档保存在同一目录
5. 每次更新后保存文件，确保状态持久化
6. 恢复执行时首先读取执行状态文件
7. 决策记录有助于后续复盘和问题追溯
