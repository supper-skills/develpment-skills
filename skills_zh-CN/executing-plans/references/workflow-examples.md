# 工作流程示例

本文档提供 executing-plans 技能的详细工作流程示例。

---

## 示例一：标准执行

```
用户：执行 design/plans/2024-01-15-auth.md 中的计划

代理：
1. 声明："我正在使用 executing-plans 技能来实施此计划。"
2. 【执行阶段 1/3】正在执行：加载并审阅计划
3. 从 design/plans/2024-01-15-auth.md 加载计划
4. 批判性审阅计划 - 未发现疑虑
5. 创建执行状态文件
6. 创建包含 8 个任务的待办列表
7. 【执行阶段 2/3】正在执行：执行任务
8. 执行每个任务：
   - 任务 1：编写测试 → 运行 → 失败
   - 任务 1：编写代码 → 运行 → 通过
   - 任务 2：编写测试 → 运行 → 失败
   - ...（继续）
9. 【执行阶段 3/3】正在执行：完成与报告
10. 所有任务完成
11. 运行最终验证
12. 报告完成并附带摘要
```

## 示例二：恢复执行

```
用户：继续执行计划

代理：
1. 声明："我正在使用 executing-plans 技能来继续实施此计划。"
2. 读取执行状态文件：design/plans/2024-01-15-auth-execution-state.md
3. 【执行阶段 2/3】正在执行：执行任务
   当前处理：任务 3（共 8 个任务）
4. 执行任务 3
5. 测试意外失败
6. 停止并报告：
   "在任务 3 被阻碍：测试需要 'lodash' 但未安装。
    我应该：
    a) 将 lodash 添加到依赖项
    b) 等待您安装
    c) 修改方法"
7. 更新执行状态文件记录问题
8. 等待指导
```

## 示例三：计划疑虑

```
用户：执行 plans/feature-x.md 中的计划

代理：
1. 声明："我正在使用 executing-plans 技能来实施此计划。"
2. 【执行阶段 1/3】正在执行：加载并审阅计划
3. 加载计划
4. 审阅并发现疑虑：
   - 文件路径引用不存在的目录
   - 测试命令使用错误的框架
5. 更新执行状态文件记录疑虑
6. 停止并询问用户：
   "我在计划中发现问题：
    - 路径 'src/lib/' 不存在。应该是 'src/utils/' 吗？
    - 计划使用 'jest' 但项目使用 'vitest'
    我应该继续修正还是等待计划修订？"
7. 等待用户指导
```

## 示例四：多子系统执行

```
用户：执行 design/plans/2024-01-15-feature-outline.md 中的计划

代理：
1. 声明："我正在使用 executing-plans 技能来实施此多子系统计划。"
2. 读取纲领文件：design/plans/2024-01-15-feature-outline.md
3. 识别到 3 个子系统：A（无依赖）、B（依赖 A）、C（依赖 A、B）
4. 创建顶层执行状态文件
5. 创建子系统 A 状态文件

子系统 A 执行：
6. 【执行阶段 0/3】正在执行：多子系统处理
   - 子系统 A：进行中
7. 【执行阶段 1/3】正在执行：加载并审阅计划（子系统 A）
8. 读取子系统 A 计划：design/plans/2024-01-15-subsystem-a.md
9. 批判性审阅计划 - 未发现疑虑
10. 【执行阶段 2/3】正在执行：执行任务（子系统 A）
11. 执行子系统 A 任务 1-3
12. 【执行阶段 3/3】正在执行：完成与报告（子系统 A）
13. 标记子系统 A 完成
14. 更新顶层状态文件：子系统 A 已完成

子系统 B 执行（依赖 A）：
15. 检查依赖：子系统 A 已完成
16. 【执行阶段 0/3】正在执行：多子系统处理
   - 子系统 B：进行中
17. 【执行阶段 1/3】正在执行：加载并审阅计划（子系统 B）
18. 读取子系统 B 计划：design/plans/2024-01-15-subsystem-b.md
19. 批判性审阅计划 - 未发现疑虑
20. 【执行阶段 2/3】正在执行：执行任务（子系统 B）
21. 执行子系统 B 任务 4-6
22. 【执行阶段 3/3】正在执行：完成与报告（子系统 B）
23. 标记子系统 B 完成
24. 更新顶层状态文件：子系统 B 已完成

子系统 C 执行（依赖 A、B）：
25. 检查依赖：子系统 A 已完成，子系统 B 已完成
26. 【执行阶段 0/3】正在执行：多子系统处理
   - 子系统 C：进行中
27. 【执行阶段 1/3】正在执行：加载并审阅计划（子系统 C）
28. 读取子系统 C 计划：design/plans/2024-01-15-subsystem-c.md
29. 批判性审阅计划 - 未发现疑虑
30. 【执行阶段 2/3】正在执行：执行任务（子系统 C）
31. 执行子系统 C 任务 7-8
32. 【执行阶段 3/3】正在执行：完成与报告（子系统 C）
33. 标记子系统 C 完成
34. 更新顶层状态文件：子系统 C 已完成

所有子系统完成：
35. 运行跨子系统集成测试
36. 生成最终报告
```

## 多子系统处理详细流程

### 0.1 识别子系统数量

- 读取纲领文件，检查子系统列表
- **单一子系统**：跳过本章，直接进入阶段一
- **多子系统**：按本章流程执行

### 0.2 验证依赖关系

- 读取纲领文件中的"子系统依赖分析"
- 按执行顺序排列子系统（不修改纲领文件）

### 0.3 创建执行状态文件

为整个项目和每个子系统创建状态文件。模板见 `execution-state-template.md`。

### 0.4 按依赖顺序执行子系统

```
对于每个子系统（按执行顺序）：
1. 检查依赖是否满足，读取依赖子系统的状态文件
2. 为该子系统创建执行状态文件
3. 进入阶段一（加载并审阅计划）
4. 进入阶段二（执行任务）
5. 标记子系统完成，更新顶层状态文件
6. 继续下一个子系统
```

### 0.5 共享组件处理

- 读取纲领文件中的"共享组件"信息
- 在第一个使用该组件的子系统中实现
- 后续子系统直接引用已实现的共享组件

### 0.6 多子系统恢复执行

1. 读取顶层执行状态文件
2. 识别中断的子系统（状态为"进行中"或"待执行"）
3. 检查每个已完成子系统的状态文件
4. 验证代码和测试是否正常
5. 验证依赖关系后继续执行

## 恢复执行详细流程

### 恢复前验证检查

| 验证项         | 命令/方法        | 目的           |
| -------------- | ---------------- | -------------- |
| Git 状态       | `git status`     | 确认工作区状态 |
| 已完成任务代码 | 检查文件是否存在 | 确认代码已写入 |
| 已完成任务测试 | 运行相关测试命令 | 确认功能正常   |
| 分支状态       | `git branch`     | 确认在正确分支 |
| 执行状态文件   | 读取状态文件     | 确认上下文正确 |

### 恢复执行选项

检测到中断后，向用户报告：

```
检测到计划执行中断：
- 计划：[计划路径]
- 状态文件：[状态文件路径]
- 已完成：[已完成数量]（共 [总数] 个任务）
- 中断位置：[中断任务名称]
- 代码验证：[验证结果]

请选择恢复方式：
a) 继续执行 - 从中断位置继续执行后续任务
b) 重新执行 - 重新执行中断的任务
c) 从头执行 - 清除进度，从头开始执行所有任务
```
