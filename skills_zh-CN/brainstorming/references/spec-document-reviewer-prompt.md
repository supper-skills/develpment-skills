## 审阅目的

确保每个规格文档在进入实施规划阶段前符合质量标准。审阅者验证完整性、一致性和对设计原则的遵循。

## 派发时机

规格文档写入 `design/specs/` 目录后。

---

## 审阅检查清单

### 1. 完整性

- [ ] 无 TODO 标记或占位符文本
- [ ] 无 "TBD" 或"待定义"章节
- [ ] 无"等 X 完成后再规格说明"的表述
- [ ] 所有章节具有适当的详细程度
- [ ] 无明显比其他章节简略的部分

### 2. 覆盖范围

- [ ] 错误处理已涵盖
- [ ] 边界情况已考虑
- [ ] 集成点已定义
- [ ] 数据流已记录
- [ ] 安全考量（如适用）

### 3. 一致性

- [ ] 无内部矛盾
- [ ] 无冲突的需求
- [ ] 术语使用一致
- [ ] 外部系统引用清晰

### 4. 清晰性

- [ ] 无模糊的需求
- [ ] 技术术语已定义或不言自明
- [ ] 需求可测试/可验证
- [ ] 验收标准明确

### 5. YAGNI（你不会需要它）

- [ ] 无未请求的功能
- [ ] 无过度工程
- [ ] 无投机性的未来预留
- [ ] 范围聚焦于已声明的需求

### 6. 范围验证

- [ ] 聚焦程度足以制定单一实施计划
- [ ] 未覆盖多个独立子系统
- [ ] 如有多个子系统，已建议拆分
- [ ] 每个子系统可产出可运行、可测试的文件

### 7. 架构质量

- [ ] 单元具有清晰边界
- [ ] 接口定义明确
- [ ] 每个单元可独立理解
- [ ] 每个单元可独立测试
- [ ] 无需阅读内部实现即可理解每个单元的功能
- [ ] 内部实现可变更而不影响使用者

## 需特别关注的问题

### 占位符内容

❌ 错误示例：

```
## 错误处理
TODO: 定义错误处理策略
```

✅ 正确示例：

```
## 错误处理
- 网络错误：指数退避重试（最多3次）
- 验证错误：返回400及详细错误信息
- 认证错误：返回401并重定向至登录页
```

### 模糊需求

❌ 错误示例：

```
系统应该快速且安全。
```

✅ 正确示例：

```
系统应在200ms内响应API请求（p95）。
所有API端点需要通过JWT令牌进行认证。
```

### 边界缺失

❌ 错误示例：

```
UserService负责用户注册、认证、
个人资料管理、通知和分析。
```

✅ 正确示例：

```
UserService负责用户注册和认证。
- ProfileService：负责个人资料管理
- NotificationService：负责通知
- AnalyticsService：负责分析
```

### 过度工程

❌ 错误示例：

```
我们将实现一个插件系统，以允许未来扩展
我们可能在2-3年后需要的功能。
```

✅ 正确示例：

```
我们将实现所请求的具体功能。
扩展点可在实际需求出现时添加。
```

## 审阅输出格式

审阅后，按以下格式提供输出：

```markdown
## 规格审阅

**状态：** ✅ 已批准 | ❌ 发现问题

**发现的问题（如有）：**

1. [章节名称]：[具体问题]
   - 为何重要：[解释]
   - 建议：[如何修复]

**优点：**

- [做得好的地方]

**建议（仅供参考）：**

- [不阻碍批准的建议]
```

---

## 审阅指导

1. **全面但公正** - 聚焦于会阻碍实施的问题
2. **解释原因** - 标记问题时，解释为何重要
3. **建设性** - 提供具体建议
4. **考虑上下文** - 规格是给将创建实施计划的工程师看的
5. **重视清晰** - 模糊的规格导致实施失败
6. **执行边界** - 不清晰的边界导致代码纠缠

---

## 升级处理

如果同一规格的审阅循环超过5次迭代：

1. 汇总所有迭代中发现的问题
2. 记录已尝试的解决方案
3. 识别具体的分歧点
4. 提交人类寻求指导，附带完整上下文

## 审阅示例

### 输入规格

```markdown
# 用户认证设计

## 概述

为应用添加用户认证功能。

## 需求

- 用户可以登录
- 用户可以登出
- 会话是安全的

## 架构

AuthService将处理所有与认证相关的事情。
```

### 审阅输出

```markdown
## 规格审阅

**状态：** ❌ 发现问题

**发现的问题：**

1. **概述**：过于模糊 - 缺少当前系统状态的上下文
   - 为何重要：实施方法取决于现有架构
   - 建议：添加章节描述当前认证状态（无？遗留系统？）

2. **需求**：缺少关键细节
   - 为何重要："登录"可能意味着多种方式 - 密码、SSO、OAuth？
   - 建议：明确认证方式（密码+邮箱、OAuth提供商等）

3. **需求**："会话是安全的"不可测试
   - 为何重要：无法验证需求是否满足
   - 建议：定义具体安全措施（HTTPS、令牌过期、CSRF防护）

4. **架构**：AuthService边界不清
   - 为何重要："所有与认证相关的事情"范围太广
   - 建议：拆分为专注的服务（TokenService、SessionService、PasswordService）

**优点：**

- 概述中目标清晰

**建议（仅供参考）：**

- 考虑添加错误处理章节
- 考虑添加测试策略章节
```
