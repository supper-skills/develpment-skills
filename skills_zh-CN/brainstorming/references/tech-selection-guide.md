# 技术选型指导

本文档提供系统性的技术选型评估框架，供头脑风暴阶段按需查阅。

---

## 评估维度

| 维度 | 考量要点 | 评估问题 |
|------|----------|----------|
| 团队熟悉度 | 团队经验、学习曲线 | 团队是否已有相关经验？学习成本如何？ |
| 社区生态 | 文档质量、社区活跃度 | 文档是否完善？遇到问题能否找到解决方案？ |
| 长期维护 | 项目活跃度、版本更新 | 项目是否持续维护？向后兼容性如何？ |
| 性能特性 | 性能表现、已知瓶颈 | 是否满足性能需求？有无已知问题？ |
| 集成难度 | 兼容性、迁移成本 | 与现有系统的兼容性如何？迁移成本多大？ |

---

## 选型流程

```
1. 明确技术约束条件
   ├── 团队技能储备
   ├── 现有技术栈
   └── 预算/时间限制

2. 列出候选技术方案（通常 2-3 个）

3. 按评估维度进行对比分析

4. 给出推荐方案及理由

5. 记录选型决策及权衡考量
```

## 决策记录模板

在规格文档中记录技术选型决策：

```markdown
## 技术选型决策

### 候选方案
| 方案 | 团队熟悉度 | 社区生态 | 长期维护 | 性能特性 | 集成难度 |
|------|-----------|----------|----------|----------|----------|
| 方案A | 高 | 高 | 高 | 中 | 低 |
| 方案B | 中 | 高 | 高 | 高 | 中 |
| 方案C | 低 | 中 | 中 | 高 | 高 |

### 选择方案A的理由
- 团队已有经验，学习成本低
- 与现有系统兼容性好
- 社区活跃，问题易解决

### 权衡考量
- 性能略低于方案B，但满足需求
- 牺牲部分性能换取开发效率
```

## 常见陷阱

| 陷阱 | 表现 | 应对策略 |
|------|------|----------|
| 追逐流行 | 仅因"流行"而选择新技术 | 评估实际需求匹配度 |
| 忽视学习成本 | 低估团队学习时间 | 预留学习缓冲期 |
| 缺乏维护支持 | 选择停止维护的项目 | 检查项目活跃度指标 |
| 过度设计 | 选择过于复杂的方案 | 遵循 YAGNI 原则 |

- [ ] 团队是否有相关经验？
- [ ] 文档是否完善？
- [ ] 社区是否活跃（GitHub stars、issues、更新频率）？
- [ ] 是否满足性能需求？
- [ ] 与现有系统兼容性如何？
- [ ] 迁移成本是否可接受？
- [ ] 是否有长期维护保障？
- [ ] 是否存在已知的重大问题？
