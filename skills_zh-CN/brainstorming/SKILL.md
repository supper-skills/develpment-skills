# 头脑风暴技能

本技能是任何创造性工作前的必经步骤，通过协作对话将创意转化为完整的设计规格说明。

---

## 核心约束

> ⚠️ 以下规则必须始终遵守，不可遗忘。

### 禁止行为（直到获得用户批准）

在用户批准设计文档前，**绝对禁止**：

- ❌ 调用任何实施技能
- ❌ 编写任何代码
- ❌ 搭建项目框架
- ❌ 采取任何实施行动

**无例外**：无论项目看似多简单，都必须遵循此流程。

### 终止条件

本技能的**唯一终止方式**是调用 `writing-plan` 技能，并传递规格文档路径。

### 开始声明

激活技能后，必须立即声明：

> "我正在使用 brainstorming skill 来探索您的需求并创建设计规格说明。"

### 阶段声明

每轮对话开始时，必须声明当前进度：

> 【阶段 X/8】正在执行：[阶段名称]

---

## 状态驱动流程

> 📍 每次对话只关注当前阶段，完成后进入下一阶段。

### 状态文件机制

**目的**：解决多轮对话中的注意力丢失问题。

**操作**：技能激活后，立即创建状态文件：

- **路径**：`design/specs/YYYY-MM-DD-<topic>-state.md`
- **模板**：参见 `references/state-template.md`

**每轮对话开始时**：读取状态文件 → 恢复上下文 → 声明当前阶段

**每次获得用户确认后**：更新状态文件 → 检查是否可进入下一阶段

---

### 阶段 1/8：探索项目背景

**目标**：了解当前项目状态，评估需求范围。

**执行动作**：

1. 检查项目文件结构
2. 查看相关文档
3. 检查近期提交记录（如有版本控制）
4. 评估需求是否涉及多个独立子系统

**范围检查**：

- 若请求涉及多个独立子系统，**立即标记**
- 协助用户分解为子项目，每个子项目独立进行头脑风暴

**完成标准**：已了解项目技术栈和现有结构，已确认需求范围是否适当

---

### 阶段 2/8：提出澄清性问题

**目标**：逐条理解目标、约束条件和成功标准。

**执行规则**：

- 每条消息**只提一个问题**
- 优先使用选择题形式
- 开放式问题也可接受

**核心问题清单**（按顺序询问）：

| 序号 | 问题类型 | 示例问题                         |
| ---- | -------- | -------------------------------- |
| 1    | 目标定位 | "这个功能的主要用户是谁？"       |
| 2    | 约束条件 | "有没有时间或技术栈的限制？"     |
| 3    | 成功标准 | "如何判断这个功能是否成功交付？" |
| 4    | 边界情况 | "有没有需要特别处理的边界情况？" |

**完成标准**：目标定位明确、约束条件清晰、成功标准可验证

---

### 阶段 3/8：提出方案选项

**目标**：提供 2-3 种不同方案及其权衡分析。

**执行动作**：

1. 提出 2-3 种不同方案
2. 说明每种方案的优缺点
3. 给出推荐方案及理由

**呈现格式**：

```markdown
## 方案对比

### 方案 A：[名称]（推荐）

- **优点**：...
- **缺点**：...
- **推荐理由**：...

### 方案 B：[名称]

- **优点**：...
- **缺点**：...
```

**完成标准**：用户已选择一个方案

---

### 阶段 4/8：呈现设计方案

**目标**：分章节展示设计，每部分获得用户批准。

**执行规则**：

- 根据复杂度调整各章节篇幅
- 简单内容用几句话概括
- 复杂细节可展开至 200-300 字
- 每完成一个部分后询问是否合适

**必须涵盖的章节**：

| 章节     | 内容要求             |
| -------- | -------------------- |
| 概述     | 功能目标、背景上下文 |
| 架构     | 系统结构、组件划分   |
| 数据流   | 数据如何流转         |
| 错误处理 | 异常情况处理策略     |
| 测试策略 | 如何验证功能正确性   |

**设计原则**：

- 将系统拆分为更小的单元
- 每个单元有明确单一的功能
- 通过清晰的接口通信
- 可独立理解和测试

**完成标准**：所有章节已呈现，用户已批准整体设计

---

### 阶段 5/8：撰写设计文档

**目标**：将验证通过的设计写入规格文档。

**执行动作**：

1. 创建规格文档：`design/specs/YYYY-MM-DD-<topic>-design.md`
2. 包含所有已确认的内容
3. 确保无 TODO、TBD 或占位符

**文档模板**：参见 `references/spec-template.md`

**完成标准**：文档已写入指定路径，无占位符内容

---

### 阶段 6/8：规格审阅循环

**目标**：通过子代理审阅确保规格质量。

**执行动作**：

1. 使用 `references/spec-document-reviewer-prompt.md` 中的审阅标准
2. 派遣审阅子代理检查规格文档
3. 若发现问题：修复并重新审阅
4. 最多 5 次迭代，之后转交人工处理

**审阅重点**：

- 完整性：无 TODO/TBD/占位符
- 一致性：无内部矛盾
- 清晰性：需求可测试/可验证
- YAGNI：无未请求的功能
- 边界清晰：单元可独立理解

**完成标准**：审阅状态为 ✅ 已批准

---

### 阶段 7/8：用户审阅

**目标**：获得用户对规格文档的最终批准。

**执行动作**：

1. 呈现规格文档路径给用户
2. 请用户审阅并确认

**标准话术**：

> "规格文档已编写并提交至 `<path>`。请审阅，并在我们开始制定实施计划前告知是否需要修改。"

**处理用户反馈**：

- 若用户要求修改：修改后重新执行阶段 6（规格审阅循环）
- 若用户批准：进入阶段 8

**完成标准**：用户明确表示批准

---

### 阶段 8/8：转入实施阶段

**目标**：调用 writing-plan 技能制定实施方案。

**执行动作**：

1. 调用 `writing-plan` 技能
2. 传递规格文档路径

**调用格式**：

```txt
调用 writing-plan 技能，规格文档路径：<path>
```

**终止**：此阶段完成后，本技能结束。

## 关键原则速查

| 原则     | 说明                           |
| -------- | ------------------------------ |
| 一次一问 | 每条消息只提一个问题           |
| 方案对比 | 确定方案前必须提出 2-3 种选项  |
| 渐进验证 | 每部分设计需获得批准后再继续   |
| 状态持久 | 使用状态文件防止注意力丢失     |
| YAGNI    | 从设计中剔除不必要的功能       |
| 唯一出口 | 只能通过调用 writing-plan 终止 |
