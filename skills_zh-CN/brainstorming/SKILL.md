# 头脑风暴技能

本技能是任何创造性工作前的必经步骤。通过协作对话，将创意转化为完整的设计规格说明。

## 触发条件

当检测到以下情况时，主动激活此技能：

- 用户提及"新功能"、"构建"、"创建组件"
- 用户提及"添加功能"、"实现"、"开发"
- 开始任何创建或修改行为的开发工作
- 用户提供需求并希望在实施前进行规划
- 任何涉及创建新内容的请求

---

## 核心原则

### 开始声明

始终以声明开始：

> "我正在使用 brainstorming skill 来探索您的需求并创建设计规格说明。"

### 关键规则

**在提交设计并获得用户批准前，禁止：**

- 调用任何实施技能
- 编写任何代码
- 搭建项目框架
- 采取任何实施行动

此规则适用于所有项目，无论其表面复杂度如何。

### 反面案例

> "这个太简单了，不需要设计"

所有项目都必须遵循此流程。待办清单、单一功能工具、配置变更——无一例外。"简单"项目往往因未经审视的假设造成最大资源浪费。设计文档可以简短（真正简单的项目仅需几句话），但必须提交设计并获得批准。

---

## 检查清单

必须为以下每项创建任务并按顺序完成：

1. **探索项目背景** — 检查文件、文档、近期提交记录
2. **提出澄清性问题** — 逐条理解目标/限制条件/成功标准
3. **提出2-3种方案** — 包含权衡利弊及你的推荐
4. **呈现设计** — 按复杂度分章节展示，每部分均需获得用户批准
5. **撰写设计文档** — 保存至 `design/specs/YYYY-MM-DD-<topic>-design.md`
6. **规范审阅循环** — 调度规范文档审阅子代理；修复问题并重新调度直至通过（最多5次迭代，之后转交人工处理）
7. **用户审阅书面规范** — 请用户在继续前审阅规范文件
8. **转入实施阶段** — 调用 writing-plan 技能制定实施方案

**终端状态是调用 writing-plan，切勿调用其它技能。**

## 流程

### 理解核心理念

- 首先查看当前项目状态（文件、文档、最近提交记录）
- 在询问具体细节前，先评估需求范围：若请求涉及多个独立子系统（例如"构建一个包含聊天、文件存储、计费和分析功能的平台"），请立即标记此情况。切勿花费时间细化一个需要先分解的项目细节。
- 如果项目规模过大，无法用单一规格说明涵盖，应协助用户将其分解为子项目：哪些是独立模块？它们之间如何关联？应按什么顺序构建？随后按常规设计流程对第一个子项目进行头脑风暴。每个子项目都将遵循独立的规格说明→规划→实施循环。
- 对于范围适当的项目，逐一提问以完善想法
- 尽量使用选择题形式，但开放式问题也可以接受
- 每条消息仅包含一个问题——若需深入探讨某个主题，请拆分为多个问题
- 重点关注：目标定位、约束条件和成功标准的理解

### 探索方法

- 提出2-3种不同方案并说明其权衡点
- 以对话形式呈现选项，附带推荐意见及推理过程
- 优先推荐首选方案并解释理由

### 呈现设计方案

- 确认理解构建目标后展示设计思路
- 根据复杂度调整各章节篇幅：简单内容用几句话概括，复杂细节可展开至200-300字
- 每完成一个部分后询问当前内容是否合适
- 涵盖范围：架构、组件、数据流、错误处理、测试
- 如果有什么地方不明白，随时准备回头澄清

### 隔离与清晰性设计

- 将系统拆分为更小的单元，每个单元都有明确单一的功能，通过定义清晰的接口进行通信，并能独立理解和测试
- 对于每个单元，你应当能回答：它的作用是什么？如何使用它？它依赖哪些组件？
- 能否在不阅读内部实现的情况下理解某个单元的功能？能否在不影响使用者的情况下修改其内部实现？如果答案是否定的，说明边界划分有待改进。
- 更小、边界清晰的单元也更便于操作——你能更清晰地理解能一次性在脑海中构建上下文的代码，且当文件功能聚焦时，修改也会更可靠。文件体积过大通常意味着其承载了过多职责。

### 在现有代码库中工作时

- 在提出修改前，先了解现有结构。遵循既有模式。
- 若现有代码存在影响工作的问题（例如文件过大、边界不清、职责混乱），应在设计方案中包含针对性改进——正如优秀开发者优化所维护代码的方式。
- 避免提出无关的重构。始终聚焦于服务当前目标的内容。

### 项目初始化场景

当用户需要从零开始创建新项目时，需额外关注以下方面：

**项目结构规划：**
- 确认项目类型（Web 应用、CLI 工具、库/SDK、微服务等）
- 推荐合适的项目目录结构
- 确定包管理器和构建工具

**基础设施决策：**
- 版本控制策略（Git 分支模型、提交规范）
- CI/CD 需求评估
- 容器化需求（Docker、Kubernetes）

**关键问题清单：**
- 项目的主要目标用户是谁？
- 预期的项目规模和复杂度？
- 是否需要支持多环境（开发、测试、生产）？
- 是否有合规性或安全性要求？

### 技术选型指导

在确定技术方案时，需系统性地评估技术选型：

**评估维度：**

| 维度 | 考量要点 |
|------|----------|
| 团队熟悉度 | 团队是否已有相关经验？学习曲线如何？ |
| 社区生态 | 文档质量、社区活跃度、问题解决资源 |
| 长期维护 | 项目活跃度、版本更新频率、向后兼容性 |
| 性能特性 | 是否满足性能需求？有无已知瓶颈？ |
| 集成难度 | 与现有系统的兼容性、迁移成本 |

**选型流程：**
1. 明确技术约束条件（团队技能、现有技术栈、预算限制）
2. 列出候选技术方案（通常 2-3 个）
3. 按评估维度进行对比分析
4. 给出推荐方案及理由
5. 记录选型决策及权衡考量

**避免常见陷阱：**
- 不要仅因"流行"而选择新技术
- 不要忽视团队的学习成本
- 不要选择缺乏长期维护支持的方案
- 不要过度设计技术架构

### 风险评估

在设计阶段必须识别和评估潜在风险：

**风险类别：**

| 类别 | 示例风险 |
|------|----------|
| 技术风险 | 技术方案未经验证、依赖项不稳定、性能瓶颈 |
| 范围风险 | 需求蔓延、功能边界模糊、依赖外部系统 |
| 资源风险 | 人力不足、时间紧迫、技能缺口 |
| 集成风险 | 第三方 API 变更、数据迁移复杂、兼容性问题 |
| 安全风险 | 数据泄露、认证漏洞、权限控制缺陷 |

**风险评估流程：**
1. **识别风险** — 列出所有可能影响项目的风险因素
2. **评估影响** — 按严重程度分级（高/中/低）
3. **评估概率** — 估计发生可能性（高/中/低）
4. **制定缓解策略** — 针对高影响/高概率风险制定应对方案
5. **记录风险清单** — 在设计文档中包含风险章节

**风险缓解策略示例：**

- **技术验证**：对不确定的技术方案进行原型验证
- **增量交付**：将大功能拆分为可独立交付的小功能
- **备选方案**：为关键依赖准备替代方案
- **监控预警**：建立关键指标的监控机制

## 设计完成之后

- 将验证通过的设计（规范）写入 `design/specs/YYYY-MM-DD-<topic>-design.md`

### 规格审查循环

编写规格文档后：

- 派遣规格文档审查子代理（参见 [spec-document-reviewer-prompt](./references/spec-document-reviewer-prompt.md)）
- 若发现问题：修复、重新派遣、重复直至批准
- 若循环超过5次，提交给人类寻求指导

### 用户审查关卡

规格审查循环通过后，在继续之前请用户审查已编写的规格：

> "规格文档已编写并提交至 `<path>`。请审阅，并在我们开始制定实施计划前告知是否需要修改。"

等待用户回应。若用户要求修改，则进行修改并重新运行规格审查循环。仅在用户批准后继续。

### 实施

- 调用 writing-plan 技能，并告知 规则文档路径 `<path>`，制定详细的实施方案
- 请勿调用任何其他技能。撰写计划是下一步。

---

## 关键原则

- **一次只提一个问题** - 避免用多个问题造成压力
- **优先选择题** - 在可行情况下，比开放式问题更易于回答
- **严格遵循YAGNI原则** - 从所有设计中剔除不必要的功能
- **探索替代方案** - 在确定方案前始终提出2-3种方法
- **渐进式验证** - 展示当前设计，获得批准后再继续推进
- **保持灵活** - 遇到不理解的内容时及时回溯澄清

## 工作流程示例

```
用户：我需要为我的应用添加用户认证系统。

代理：
1. 声明："我正在使用 brainstorming 技能来探索您的需求并创建设计规格说明。"
2. 探索项目背景 - 检查现有代码结构、技术栈
3. 逐一提出澄清问题：
   - "您希望支持哪些认证方式？"
   - "需要社交登录集成吗？"
   - "会话管理有什么要求？"
4. 提出2-3种方案及其权衡
5. 分章节呈现设计，每部分获得批准
6. 将设计文档写入 design/specs/2024-01-15-user-auth-design.md
7. 派遣规格文档审查子代理，修复任何问题
8. 请用户审阅规格
9. 获得批准后，调用 writing-plan 技能
```

---

## 注意事项

1. **始终在开始时声明技能使用**
2. **头脑风暴期间不编写代码或实施**
3. **每条消息一个问题** - 避免压倒性信息
4. **始终在确定方案前提出替代方案**
5. **渐进式获得批准** - 逐章节进行
6. **用户审阅前先进行规格审阅** - 提前发现问题
7. **过渡到规划前需要用户批准**
8. **最后一步只调用 writing-plan**
